# Session 018 — 2026-03-13

## Summary

Blueprint-to-session traceability quick fix. The Blueprint Workbench and Registry detail page now render a "← Intake Session" link that navigates back to the originating intake conversation. Previously, once a designer or compliance officer landed on a blueprint, there was no path back to the intake session that produced it.

## Actions

### Blueprint Workbench (`src/app/blueprints/[id]/page.tsx`) — Modified

Added `const [sessionId, setSessionId] = useState<string | null>(null)` state variable.

In the `fetch(`/api/blueprints/${id}`)` callback, added:
```typescript
if (data.sessionId) setSessionId(data.sessionId as string);
```
`sessionId` is already present in the full blueprint row returned by `GET /api/blueprints/[id]` — no API change needed.

Added "← Intake Session" link in the header's right-side button group (before "View in Registry →"), conditional on `sessionId` being set:
```tsx
{sessionId && (
  <Link href={`/intake/${sessionId}`} ...>
    ← Intake Session
  </Link>
)}
```

### Registry Detail Page (`src/app/registry/[agentId]/page.tsx`) — Modified

Added `sessionId: string | null` to the `BlueprintVersion` interface.

`GET /api/registry/[agentId]` returns `agent: latest` (full DB row), which already includes `sessionId` — no API change needed.

Added "← Intake Session" link in the header's right-side button group (before "Open in Studio"), conditional on `latest.sessionId`:
```tsx
{latest.sessionId && (
  <Link href={`/intake/${latest.sessionId}`} ...>
    ← Intake Session
  </Link>
)}
```
