# Session 016 — 2026-03-13

## Summary

Phase 14: MRM Report HTML View. Compliance officers and administrators can now open a full, browser-rendered MRM Compliance Report from the Registry detail page and print it to PDF. The JSON download is retained as a machine-readable artifact; both formats are audited identically.

## Actions

### Phase 14: MRM Report HTML View

#### 1. `src/components/mrm/print-button.tsx` — NEW
Thin `"use client"` component that calls `window.print()` on click. Isolated from the server component report page to avoid making the page a client component. Has `print:hidden` class so it disappears on print.

#### 2. `src/app/blueprints/[id]/report/page.tsx` — NEW
Async server component that assembles and renders the full MRM Compliance Report as structured HTML.

**Auth + access:**
- Requires `compliance_officer` or `admin` role — renders an "Access denied" page for other roles.
- Calls `assertEnterpriseAccess` on the blueprint before assembling (prevents expensive assembly for cross-enterprise requests).
- Writes a `blueprint.report_exported` audit entry with `metadata.format: "html"` on every successful render — fully auditable alongside JSON exports.

**Rendering (11 sections):**
- **Cover**: Agent name, status, version badge, enterprise, generated-at/by, blueprint + agent IDs.
- **Section 2 — Risk Classification**: Color-coded risk tier chip (red/amber/green), tier basis narrative, intended use, business owner, model owner, deployment type, data sensitivity, regulatory scope, stakeholders consulted.
- **Section 3 — Agent Identity**: Name, description, persona, tags.
- **Section 4 — Capabilities**: Stats (tool count, knowledge source count, instructions configured) + tools table + knowledge sources table.
- **Section 5 — Governance Validation**: Valid/invalid status, policy/error/warning counts, violation list with severity badges and remediation suggestions.
- **Section 6 — Review Decision**: Color-coded outcome chip + reviewer + timestamp + comment.
- **Section 7 — SOD Evidence**: Designer / reviewer / deployer identities + SOD satisfied status (green ✓ / red ✗).
- **Section 8 — Deployment Record**: Deployed status, deployed at/by, change reference, notes.
- **Section 9 — Model Lineage**: Version history table + deployment lineage table (all versions).
- **Section 10 — Audit Chain**: Chronological event table (timestamp, action, actor, role, transition).
- **Section 11 — Stakeholder Contributions**: Coverage gap callout + per-domain contribution cards showing all fields.

**Print / PDF:**
- `PrintButton` calls `window.print()` — no library dependency.
- Toolbar and print button use `print:hidden` Tailwind variant to disappear on print.
- Section headers use `print:break-before-page` to produce clean page breaks in PDF output.
- Footer auto-appended: report metadata + SR 11-7 disclaimer.

#### 3. `src/app/registry/[agentId]/page.tsx` — Modified
Added "View MRM Report" `<Link>` button next to the existing JSON export button for `compliance_officer` + `admin` roles. Renamed "Export MRM Report" → "Export JSON" to distinguish between the two formats clearly.
