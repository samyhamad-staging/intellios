# Session 006 тАФ 2026-03-13

## Goal

Post-MVP Phase 3: Event-driven workflow notifications + SLA monitoring.

Strategic rationale: The pull-based review loop breaks in enterprise use. Reviewers must log in to discover work; designers have no visibility into review outcomes. Under SR 11-7 / DORA model governance requirements, timely, documented review is a compliance obligation тАФ not just a UX preference. An event-driven notification layer addresses this gap with minimal architectural overhead while opening a clean extension path for future consumers (webhooks, audit enrichment, analytics).

## Summary

**Event-Driven Notification Layer** тАФ 13 new files, 3 modified:

**Event Bus** (`src/lib/events/`): Lightweight in-process event bus. `types.ts` defines `LifecycleEvent` and `EventType`. `bus.ts` provides `registerHandler()` + fire-and-forget `dispatch()`. Handlers are isolated тАФ one failure does not block others.

**Audit-as-Event-Source**: `writeAuditLog` is the single integration point. After every DB insert, it dispatches a `LifecycleEvent` with the audit row's ID as the correlation ID. No routes need to know about notifications.

**Notification System** (`src/lib/notifications/`):
- `store.ts` тАФ CRUD against the `notifications` DB table
- `recipients.ts` тАФ role-based recipient lookup scoped to `enterpriseId`
- `email.ts` тАФ Resend email delivery, no-ops when API key absent
- `handler.ts` тАФ Routing logic for all lifecycle events тЖТ correct recipients; self-registered via side-effect import in `audit/log.ts`

**Notifications Table** тАФ new migration `0005_notifications.sql` with `notifications` table and two indexes optimized for the two primary query patterns (user inbox, enterprise audit).

**Notifications API** (`/api/notifications`) тАФ GET (list + unread count) + PATCH (mark all read).

**NotificationBell UI** тАФ client component in nav; 30s focus-aware polling; unread count badge; dropdown with relative timestamps, type icons, and inline read state.

**SLA Monitoring** (`src/lib/sla/config.ts`) тАФ `getSlaStatus()` utility with 48h warn / 72h alert thresholds, environment-variable overridable. Applied to Pipeline Board: amber ring + badge at warn, red ring + badge at alert. No background jobs required.
