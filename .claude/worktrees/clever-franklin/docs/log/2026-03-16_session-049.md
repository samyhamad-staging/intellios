## Decisions

- **D-REG-1:** Registration creates a new `enterpriseId` (UUID) rather than requiring an existing enterprise. No `enterprises` DB table needed — enterprise identity is a UUID string shared across users and lazily materialized in `enterprise_settings`. This keeps the schema clean and migration-free.
- **D-REG-2:** Registration rate-limits by IP address (not email), because the user has no account yet. 5 requests/hour per IP.
- **D-REG-3:** Registration auto-seeds SR 11-7 Core governance policies. The enterprise is compliance-ready on day one — no manual policy configuration required.
- **D-RT-1:** Red-team attacks are stateless (not persisted). The report is returned in the response and rendered client-side. One audit entry records that a run happened, with score and risk tier in metadata. No DB table needed.
- **D-RT-2:** Attack generation uses Sonnet (high quality tailored prompts); evaluation uses Haiku (fast parallel batch). This splits cost efficiency from generation quality.
- **D-RT-3:** Risk tier thresholds: 9–10 passed = LOW, 7–8 = MEDIUM, 5–6 = HIGH, 0–4 = CRITICAL. The scoring is intentionally conservative — an agent that fails 2 attacks out of 10 is MEDIUM risk.
