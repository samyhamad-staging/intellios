# Session 013 — 2026-03-13

## Summary

Phase 10: Governance Policy Management. Compliance officers and admins can now create, edit, and delete enterprise governance policies through the UI. PATCH + DELETE API endpoints added for individual policies; POST updated to allow compliance_officer role. Shared `PolicyForm` component with interactive rule builder (11 operators, 2 severity levels, per-rule ABP field + violation message). Governance Hub updated with "New Policy" CTA, Edit/View links per policy card, and improved empty state.

## Actions

### 1. `src/app/api/governance/policies/[id]/route.ts` — NEW

Added three handlers for individual policy management:

- **GET** — fetches a single policy; compliance_officer + admin; compliance_officer may not access another enterprise's policy
- **PATCH** — partial update (`name?`, `type?`, `description?`, `rules?`); compliance_officer cannot modify global (platform-level) policies or other enterprises' policies; admin can modify any policy
- **DELETE** — removes a policy; same access control rules as PATCH

All handlers:
- Authenticate with `requireAuth(["compliance_officer", "admin"])`
- Return 404 if policy does not exist
- Return 403 with descriptive message if access denied by enterprise scoping rule

### 2. `src/app/api/governance/policies/route.ts` — Modified

Changed `requireAuth(["admin"])` to `requireAuth(["compliance_officer", "admin"])` in the POST handler. Compliance officers can now create enterprise-scoped policies (policy is scoped to `authSession.user.enterpriseId`, identical to existing admin behavior).

### 3. `src/components/governance/policy-form.tsx` — NEW

Shared form component used by both the create and edit pages.

- **Header fields**: name (required), type (select: safety/compliance/data_handling/access_control/audit), description (optional)
- **Rule builder**: add/remove `PolicyRule` objects with per-row editing
  - Each rule: ABP field path, operator (all 11: exists/not_exists/equals/not_equals/contains/not_contains/matches/count_gte/count_lte/includes_type/not_includes_type), value (hidden for exists/not_exists), severity (error/warning), violation message
  - Operators without `value` hide the value input to reduce cognitive noise
  - `crypto.randomUUID()` for new rule IDs
  - Empty-rule handling: `emptyRule()` creates a pre-typed `PolicyRule` for new additions
- **Client-side validation**: name required; all rule fields required; value required for operators that use it; inline error messages under each failing field
- **Read-only mode**: used for platform policies in the edit page — all inputs disabled, no submit button
- **Props**: `initialValues?`, `onSubmit`, `submitLabel?`, `saving?`, `readOnly?`

### 4. `src/app/governance/policies/new/page.tsx` — NEW

Client component wrapping `PolicyForm`. POSTs to `/api/governance/policies` on submit. Redirects to `/governance` on success. Displays server-returned error message on failure.

### 5. `src/app/governance/policies/[id]/edit/page.tsx` — NEW

Client component. Fetches `/api/governance/policies/[id]` on mount and pre-populates `PolicyForm` via `initialValues`. Shows amber warning banner for platform policies (reads as informational, `readOnly` mode). Enterprise-owned policies show a delete section at the bottom with two-step confirmation ("Delete Policy" → "Yes, Delete" + "Cancel"). Uses React `use(params)` for Next.js 15 async params.

### 6. `src/app/governance/page.tsx` — Modified

- Imported `useSession` from `next-auth/react`; derived `canManagePolicies` boolean (`admin || compliance_officer`)
- Policy Library header: added "New Policy" button (blue, links to `/governance/policies/new`) visible to `canManagePolicies` users
- Empty state: replaced static "API by administrators" text with "Create first policy" CTA for managers; fallback for non-managers says "Contact your administrator"
- Policy cards: added "Edit →" (enterprise policies) or "View →" (platform policies) links in the right column, visible to `canManagePolicies` users
