# Session 089 — 2026-04-02

## Summary

Per-enterprise IdP routing / SAML SSO completion (Priority #1 from strategic roadmap). Replaced the single platform-level OIDC provider (gated by `SSO_ISSUER` env var) with dynamic per-enterprise OIDC providers loaded from the database at cold start. Each enterprise that has SSO enabled now uses its own IdP credentials (`issuer`, `clientId`, `clientSecret` stored in `enterprise_settings.sso`). The login flow routes to the correct enterprise IdP via a `providerId` returned from the `sso-check` endpoint. JIT provisioning now extracts `enterpriseId` directly from the provider ID instead of scanning all enterprises by email domain.

## Context

Strategic analysis showed the platform at ~89% completion. The single highest-priority gap was per-enterprise IdP routing — a procurement blocker because all tenants shared one global OIDC provider (set by env vars), making it impossible to onboard multiple enterprises with different IdPs simultaneously.

## Architecture decision

- **Auth.js v5 split-config pattern**: `auth.config.ts` (Edge-safe, for middleware) + `auth.ts` (Node.js, with DB access and enterprise OIDC providers). This is the canonical Auth.js v5 recommended pattern for database-backed providers.
- **Module-level top-level `await`** in `auth.ts` to load enterprise providers at cold start. Valid in ESM/Next.js 15 server-side context; runs once per serverless instance.
- **Provider ID naming**: `oidc-{enterpriseId}` — allows JIT provisioning to extract enterpriseId directly from the provider ID without an additional DB scan.
- **`SSO_ISSUER` / `SSO_CLIENT_ID` / `SSO_CLIENT_SECRET` env vars** superseded; may remain in the environment without effect.

## SAML status

SAML remains a type enum stub (`protocol: "saml"` in `EnterpriseSettings.sso`). The per-enterprise provider ID pattern (`saml-{enterpriseId}`) is ready to receive a SAML implementation using `node-saml` or BoxyHQ SAML Jackson. Deferred to a follow-up session.

## Operational note

SSO configuration changes (new enterprise enabling SSO, rotating client secret) take effect on the next serverless function cold start — typically within minutes on Vercel, or immediately after a redeployment. This should be documented in the admin SSO UI tooltip.
