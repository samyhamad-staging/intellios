## Actionable Task Queue

### SEC-20260329-001: Harden CRON_SECRET guard in GET /api/cron/review-reminders
Priority: P2
Effort: XS
File: `src/app/api/cron/review-reminders/route.ts:29-36`

**Problem:** `CRON_SECRET` is checked only when the env var is present. If the var is not set (any environment where it was never configured), the route is completely unprotected — any caller can trigger reminder emails to compliance officers and write `lastReminderSentAt` to the database. The comment "optional; skip if unset" treats the absence of the secret as intentional; in practice this is an unprotected mutation endpoint.

**Fix:**
```typescript
// Replace the optional guard block (lines 30–36) with a required check:
const cronSecret = process.env.CRON_SECRET;
if (!cronSecret) {
  // Fail closed: if the secret is not configured, refuse all requests to prevent
  // accidental exposure in environments where the var was never set.
  return NextResponse.json({ error: "Service unavailable" }, { status: 503 });
}
const authHeader = request.headers.get("authorization");
if (authHeader !== `Bearer ${cronSecret}`) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

**Verify:** Deploy without `CRON_SECRET` set → `GET /api/cron/review-reminders` should return 503. With an incorrect token → 401. With correct token → 200.

### SEC-20260329-002: Add rate limiting to POST /api/auth/forgot-password
Priority: P2
Effort: XS
File: `src/app/api/auth/forgot-password/route.ts:25`

**Problem:** The endpoint is unbounded. An attacker who knows a target's email can send unlimited POST requests, flooding the target with password-reset emails and exhausting the application's email quota. `/api/auth/register` already uses `rateLimit()` with IP-keyed limiting — `forgot-password` should follow the same pattern.

**Fix:**
```typescript
// Add after existing imports (around line 8), before parseBody call:
import { rateLimit } from "@/lib/rate-limit";

// Add as first block inside POST handler, before parseBody (line 26):
const ip =
  request.headers.get("x-forwarded-for")?.split(",")[0].trim() ?? "unknown";
const rateLimitResponse = rateLimit(ip, {
  endpoint: "forgot-password",
  max: 5,
  windowMs: 60 * 60 * 1000, // 5 requests per hour per IP
});
if (rateLimitResponse) return rateLimitResponse;
```

**Verify:** Send 6 POST requests in sequence to `/api/auth/forgot-password` from the same IP → 6th should return 429.

### SEC-20260329-003: Block RFC 1918 / loopback targets in webhook delivery
Priority: P2
Effort: S
File: `src/lib/webhooks/deliver.ts:74` (also `deliverWebhookTest` at line 164)

**Problem:** Webhook URLs are validated as `https://` at creation time (`src/app/api/admin/webhooks/route.ts:16`), but there is no check that the URL resolves to a public address. An admin could register a webhook pointing at an internal service (e.g., `https://internal-db.corp/`, `https://169.254.169.254/latest/meta-data/`, `https://kubernetes.default/`) and use the test endpoint to probe internal infrastructure. The `deliverWebhookTest` response body (up to 500 chars) is returned directly to the caller, making this exploitable for SSRF data exfiltration. Requires admin role, but cross-enterprise platform admins and any compromised admin account are in scope.

**Fix:** Add URL validation before each `fetch()` call in `deliver.ts` and `deliverWebhookTest`. Extract hostname, resolve DNS, and reject RFC 1918 / loopback ranges. Alternatively, add the validation at webhook creation time in `CreateWebhookSchema`:

```typescript
// In src/app/api/admin/webhooks/route.ts, extend CreateWebhookSchema:
url: z.string().url()
  .refine((u) => u.startsWith("https://"), { message: "Webhook URL must use HTTPS" })
  .refine((u) => {
    try {
      const host = new URL(u).hostname;
      // Reject loopback, private ranges, and link-local
      return !/^(localhost|127\.|10\.|172\.(1[6-9]|2\d|3[01])\.|192\.168\.|169\.254\.|::1|0\.0\.0\.0)/.test(host);
    } catch { return false; }
  }, { message: "Webhook URL must point to a public host" }),
```

Note: hostname-only validation does not protect against DNS rebinding. For full SSRF protection, validate the resolved IP at delivery time (Node.js `dns.lookup` before `fetch`).

**Verify:** Attempt to create a webhook with `url: "https://192.168.1.1/hook"` → should return 400. `https://webhook.site/...` → should succeed.

### SEC-20260329-007: Add rate limiting to POST /api/auth/reset-password
Priority: P3
Effort: XS
File: `src/app/api/auth/reset-password/route.ts:24`

**Problem:** The reset-password endpoint accepts unlimited requests. The 32-byte cryptographic token makes brute-force practically infeasible (2^256 search space), so this is low severity. However, unlimited submissions allow an attacker to perform timing-based reconnaissance (valid vs. expired token lookup times differ slightly under high load) and to burn server/DB resources. Consistent with the hardening applied to `/api/auth/register`.

**Fix:**
```typescript
// Add IP-based rate limit at the top of the POST handler (same pattern as register):
const ip = request.headers.get("x-forwarded-for")?.split(",")[0].trim() ?? "unknown";
const rateLimitResponse = rateLimit(ip, {
  endpoint: "reset-password",
  max: 10,
  windowMs: 60 * 60 * 1000, // 10 per hour per IP
});
if (rateLimitResponse) return rateLimitResponse;
```
