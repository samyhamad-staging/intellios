### Merge Strategy Decision — Squash Merge

**Context:** `claude/keen-pascal` was 17 commits ahead of `main`. Both branches had duplicate work (same changes appeared in both with different commit hashes due to prior cherry-pick + merge activity).

**Options considered:**
1. 3-way merge — preserves all commits but creates noisy duplicate commit history
2. Rebase — clean but would rewrite 17 commits and require force push
3. Cherry-pick — too granular for 17 commits of related work
4. **Squash merge** — flattens keen-pascal's 17 commits into 3 logical squash commits on main; resolves conflicts once; no force push; clean linear history

**Decision:** Squash merge. Conflicts in all 9 conflicting files were resolved by taking the keen-pascal version (it was always the superset). Net result: 3 commits on main covering the 3 logical units of work.
