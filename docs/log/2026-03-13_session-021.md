# Session 021 ‚Äî 2026-03-13

## Summary

Phase 18: Blueprint Workbench UX Reliability. Four targeted fixes to the Blueprint Workbench (`/blueprints/[id]`) resolving reliability issues that were invisible until specific navigation paths triggered them.

1. **Remove ABP from URL params** ‚Äî `handleGenerate` in the intake page previously base64-encoded the full ABP + ValidationReport into the redirect URL (`?abp=...&vr=...`). Large blueprints could exceed browser URL length limits. The workbench already had a full API-fetch fallback. Redirect now passes only `agentId`; the workbench always loads content from the API.

2. **`agentId` state + API hydration** ‚Äî `agentId` was previously only available from URL search params, making the Submit for Review button invisible on direct navigation (bookmark, pipeline link, registry link). Converted to `useState`, initialized from URL params when present, populated from the `GET /api/blueprints/${id}` response when not. All 6 uses of `agentId` in the workbench updated to `agentIdState`.

3. **Auto-validate after refinement** ‚Äî After a successful `POST /refine`, the workbench previously cleared `validationReport` and left the user with "Not yet validated" state ‚Äî requiring a manual "Validate now" click before submitting. The refine handler now automatically POSTs to `/validate` after refinement completes. `setRefining(false)` in the outer finally fires after the nested validation try/catch/finally, so the Apply Changes button stays disabled during validation, preventing refinement racing.

4. **Surface validation errors on explicit calls** ‚Äî `handleValidate` previously caught all errors silently (`catch { /* non-critical */ }`). Errors from explicit "Validate now" / "Re-run" button clicks are now surfaced via `setError`.
