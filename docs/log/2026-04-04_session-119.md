# Session 119 — 2026-04-04

## Summary

**UX Audit Wave 1 — Foundation (14 items)**

Comprehensive UX audit of the Intellios codebase was performed (7-phase audit: Navigation & IA, Component Library Consistency, Design Token Usage, Accessibility, Form UX, Error Handling, Copy & Messaging) producing a 41-item remediation tracker across 4 waves. This session implemented all 14 Wave 1 "Foundation" items — the quick wins addressing component duplication, design token alignment, dangerous-action safeguards, and accessibility patterns.

**Stats:** 0 new npm dependencies · 0 schema migrations · ~20 files modified · 2 new files created · 1 file deleted

---

## Actions

### W1-01 — Delete dead `ui/heading.tsx`, redirect barrel export to Catalyst

- **Problem:** `src/components/ui/heading.tsx` contained a near-duplicate of `src/components/catalyst/heading.tsx` with hardcoded `text-zinc-950` classes that bypassed the design token system. All 36 pages already imported directly from `@/components/catalyst/heading` — the `ui/` version was unreachable dead code.
- **Action:** Deleted `src/components/ui/heading.tsx`. Updated `src/components/ui/index.ts` line 12 to re-export `Heading, Subheading` from `../catalyst/heading` rather than `./heading` for backward compatibility.
- **Files:** `src/components/ui/index.ts` (modified), `src/components/ui/heading.tsx` (deleted)

### W1-02 — Standardise Catalyst heading line-heights

- **Problem:** `src/components/catalyst/heading.tsx` had slightly inconsistent line-height utilities (`text-xl/7 sm:text-lg/7` for Heading level 2 and `text-base/7` for Subheading level 2) compared to the tighter spacing used elsewhere.
- **Action:** Heading level 2: `text-xl/7 sm:text-lg/7` → `text-xl/8 sm:text-lg/6`. Subheading level 2: `text-base/7` → `text-base/6`.
- **Files:** `src/components/catalyst/heading.tsx` (modified)

### W1-03 — Thin re-exports: `ui/switch`, `ui/description-list`, `ui/divider`

- **Problem:** Three `ui/` components were full copies of their Catalyst counterparts, creating dual maintenance surfaces.
- **Action:** Replaced each file body with a single re-export pointing to the canonical `catalyst/` implementation. Existing import paths (`@/components/ui/switch`, etc.) continue to work unchanged.
- **Files:** `src/components/ui/switch.tsx`, `src/components/ui/description-list.tsx`, `src/components/ui/divider.tsx` (all modified to thin re-exports)

### W1-04 — `lifecycle-controls.tsx`: Catalyst Button + Deprecate/Reject confirmation dialogs

- **Problem:** `LifecycleControls` imported `ui/button` (legacy variant API). Destructive actions (Deprecate, Reject) had no confirmation step — a single click in the overflow menu immediately triggered an irreversible status transition.
- **Action:**
  - Added `variantToCatalystProps()` helper mapping `"primary" | "secondary" | "ghost" | "destructive"` to Catalyst Button props (`color="indigo"`, `outline`, `plain`, `color="red"`).
  - Buttons now use `{...variantToCatalystProps(action.variant)}` spread.
  - Added `DANGER_MESSAGES` map with title/description/confirm text for `deprecated` and `rejected` states.
  - Added `pendingAction` state + `handleDangerSelect()` to route dangerous actions through a confirmation dialog.
  - Added Catalyst Dialog at bottom of component rendering the confirmation prompt before executing the transition.
  - Import updated from `@/components/ui/button` to `@/components/catalyst/button`.
- **Files:** `src/components/registry/lifecycle-controls.tsx` (modified)

### W1-05 — `api-keys/page.tsx`: Revoke confirmation dialog

- **Problem:** "Revoke" button immediately called the revoke API with no confirmation — irreversible action with no safeguard.
- **Action:** Added `revokeTarget: ApiKey | null` state. Revoke button now sets `revokeTarget` instead of calling API directly. `handleRevokeConfirmed()` (renamed from `handleRevoke`) fires only after dialog confirmation. Catalyst Dialog shows key name and explains the consequence.
- **Files:** `src/app/admin/api-keys/page.tsx` (modified)

### W1-06 — `webhooks/page.tsx`: Delete confirmation dialog

- **Problem:** `WebhookCard` delete had a two-step inline button pattern (`confirmDelete` state toggling between "Delete" and "Confirm Delete" on the same button) — easily misclicked, not a modal pattern, no description of consequences.
- **Action:** Replaced inline two-step with proper Catalyst Dialog. `showDeleteDialog` state drives the modal. `WebhookCard` return wrapped in `<>` fragment. Dialog shows webhook name and explains that all delivery history will be lost.
- **Files:** `src/app/admin/webhooks/page.tsx` (modified)

### W1-07 — `validation-report.tsx`: Policy name links to edit page

- **Problem:** Violation cards showed policy name as plain text with no navigation affordance. Compliance officers had no way to jump directly to a policy for remediation.
- **Action:** When `violation.policyId` is present, the policy name renders as a Next.js `Link` to `/governance/policies/${violation.policyId}/edit` with `hover:underline hover:text-primary` styling. Falls back to plain text when `policyId` is absent.
- **Files:** `src/components/governance/validation-report.tsx` (modified)

### W1-08 — `governance/page.tsx`: Semantic design tokens throughout

- **Problem:** Governance hub page used hardcoded Tailwind color classes (`bg-red-50`, `text-red-700`, `border-red-200`, `bg-green-50`, etc.) and framework/policy-type color maps that bypassed the design token system. Custom enterprise themes could not override these colors.
- **Action:** Python sed script replaced all hardcoded semantic colors with design token utilities:
  - `FRAMEWORK_COLORS` and `POLICY_TYPE_COLORS` maps → `bg-policy-{type}-subtle text-policy-{type}-text border-policy-{type}-border` tokens.
  - Error states → `border-danger-muted bg-danger-muted text-danger-text`.
  - Success states → `bg-success-muted border-success-muted text-success-text`.
  - Warning states → `bg-warning-muted text-warning`.
  - Violation counts → `text-danger`.
  - Calendar/info → `border-info-muted bg-info-muted text-info-text`.
- **Files:** `src/app/governance/page.tsx` (modified)

### W1-09 — `compliance/page.tsx`: Semantic design tokens throughout

- **Problem:** Same issue as governance page — hardcoded red/green/yellow/blue Tailwind classes throughout status displays, error banners, overdue notices, table cells, and links.
- **Action:** Same token-replacement approach. Danger, success, warning, info states all now use semantic token utilities.
- **Files:** `src/app/compliance/page.tsx` (modified)

### W1-10 — `governor-sidebar.tsx`: Design token alignment

- **Problem:** Governor sidebar used hardcoded `bg-violet-600` for avatar, `#a78bfa` hex for active icon color, and a fixed `w-56` width class — none of these could be overridden by enterprise branding or the CSS token system.
- **Action:**
  - Avatar: `bg-violet-600` → `bg-primary`.
  - Active icon color: `"#a78bfa"` → `"var(--sidebar-accent)"`.
  - Active nav item: added `boxShadow: active ? "var(--sidebar-active-glow, none)" : "none"` for theming.
  - Width: `w-56` class removed; replaced with `style={{ width: "var(--sidebar-width, 240px)" }}`.
  - Logo fallback: `"#7c3aed"` hardcoded → `"var(--color-primary)"`.
  - Added `import React from "react"` for `React.CSSProperties` type.
- **Files:** `src/components/nav/governor-sidebar.tsx` (modified)

### W1-11 — `mobile-layout.tsx`: Max-width content container

- **Problem:** Main content area had no max-width constraint — on very wide monitors (2560px+) content stretched to full viewport width, degrading readability.
- **Action:** Wrapped children in `<div className="max-w-screen-2xl mx-auto">` inside the `<main>` element.
- **Files:** `src/components/nav/mobile-layout.tsx` (modified)

### W1-12 — New error boundaries for Governor and Admin sub-apps

- **Problem:** `app/governor/` and `app/admin/` had no `error.tsx` Next.js error boundaries. Any unhandled runtime error in these sections would bubble to the root error boundary (or crash the entire app), with no contextual recovery affordance.
- **Action:** Created `src/app/governor/error.tsx` and `src/app/admin/error.tsx`. Both use `"use client"` directive, accept `{ error, reset }` props, render a semantic `bg-danger-muted` error card with Catalyst Button for retry and a back link to the respective section home.
- **Files:** `src/app/governor/error.tsx` (created), `src/app/admin/error.tsx` (created)

### W1-13 — `form-field.tsx`: Character counter + on-blur validation

- **Problem:** `FormField` showed validation errors immediately on render (before user interaction), creating noise. There was no character counter for length-bounded fields.
- **Action:**
  - Added `maxLength?: number` prop — when provided, renders a right-aligned `currentLength / maxLength` counter in the label row with three color states: normal (`text-text-secondary`), near-limit at 90% (`text-warning`), over-limit (`text-danger`).
  - Added `currentLength?: number` prop (pass `value.length`).
  - Added `touched?: boolean` prop — when provided, error only renders when `touched === true` (on-blur validation). Omitting `touched` preserves existing behavior (always show error).
  - Backward compatible — no existing callsites require changes.
- **Files:** `src/components/ui/form-field.tsx` (modified)

### W1-14 — `review-panel.tsx`: Collapsible approval history section

- **Problem:** Review panel had no way to see the history of prior review steps. Reviewers had no context about what previous reviewers had decided or commented.
- **Action:**
  - Added `ApprovalHistoryEntry` interface (`stepLabel`, `reviewerName?`, `reviewerEmail`, `action: "approve" | "request_changes" | "reject"`, `comment`, `decidedAt`).
  - Added `approvalHistory?: ApprovalHistoryEntry[]` prop to `ReviewPanelProps` (default `[]`).
  - Added `historyExpanded` toggle state with `ChevronDown/ChevronUp` icons from lucide-react.
  - Renders a collapsible "Approval History (N)" section above the Decision radio buttons. Each entry shows step label, action badge with semantic color (green/yellow/red), reviewer name/email, rationale comment, and `toLocaleDateString()` timestamp.
- **Files:** `src/components/review/review-panel.tsx` (modified)

### W1-15 — `policy-form.tsx`: UX improvements

- **Problem:** Runtime policy info note used hardcoded blue classes. "Add Rule" button used hardcoded blue. Simulation button gave no feedback when disabled due to no rules. Focus ring used `focus:border-blue-400` hardcoded.
- **Action:**
  - Runtime Policy note: `border-blue-100 bg-blue-50` → `border-info-muted bg-info-muted`; added link "View runtime violations in Governance Hub →".
  - Simulation disabled explanation added below button when `rules.length === 0`.
  - Rule error text: `text-red-600` → `text-danger`.
  - Add Rule button: `border-blue-200 bg-blue-50 text-blue-700` → `border-info-muted bg-info-muted text-info-text`.
  - Simulation result cards: semantic danger/success tokens.
  - All `focus:border-blue-400` → `focus:border-primary`.
- **Files:** `src/components/governance/policy-form.tsx` (modified)

### W1-16 — `registry/[agentId]/page.tsx`: Migrate to Catalyst Button

- **Problem:** Registry agent detail page imported legacy `ui/button` with `variant="secondary" size="md"` API, which is inconsistent with Catalyst.
- **Action:** Import updated to `@/components/catalyst/button`. Button props changed to `outline` (the Catalyst equivalent of secondary).
- **Files:** `src/app/registry/[agentId]/page.tsx` (modified)
