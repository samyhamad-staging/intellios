## Decision — Path A vs. Path B for binary hosting

The runbook documents both:

- **Path A** — self-host on the existing artifact bucket. Durable, Intellios-controlled, audit-friendly, no third-party rate-limit risk. The added publisher script automates everything except Vercel env-var paste.
- **Path B** — point `CHROMIUM_REMOTE_EXECUTABLE_URL` directly at the Sparticuz GitHub release URL. Acceptable for the first smoke; not acceptable for production because GH download outages become Intellios outages.

Recommendation in the runbook: Path A from the start. The publisher script makes the marginal cost of doing it right ~30 seconds.

If Samy elects Path B for the smoke, the runbook explicitly directs him to open a follow-up Story to migrate before any design-partner demo. Either way, ADR-015's design intent (a deterministic, infrastructure-controlled binary location) is preserved.

## Notes for the operator (Samy) on first run

1. The publisher script will fail loud if `ARTIFACT_BUCKET` is unset — that's intentional. We do not want the script to invent a bucket name.
2. The script computes the download SHA-256 and prints it. Cross-check against the GitHub release page before pasting the URL into Vercel — this is the only protection against a release-asset replacement attack.
3. If the bucket has Object Ownership = `BucketOwnerEnforced` (the modern AWS default), the `--acl public-read` will be silently ignored. The script verifies anonymous read at the end and emits a templated bucket-policy snippet on failure. Apply it, re-run the verify step manually with `curl`.
4. After Vercel env-var save, the new value is picked up on next cold start. If the route still throws "URL is not set", redeploy the existing build from the Vercel UI — that recycles all warm function instances.
