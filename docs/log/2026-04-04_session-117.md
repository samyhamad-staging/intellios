## Actions

### Continuation setup
- Resumed from previous context; identified W-02/W-11/M-05 as the in-progress item.

### W-02 / W-11 / M-05 — Review queue count mismatch
- **Root cause:** `review/page.tsx` was passing `?role=admin` to `/api/review`. The API filtered blueprints by `activeStep.role === "admin"`, so blueprints at "reviewer" steps were invisible to admin users, making the queue appear empty while the Overview count showed items awaiting review.
- **Fix:** In `src/app/review/page.tsx`, changed role param logic so admin users do NOT send a role filter (they see all in-review items regardless of approval step).

### W-05 — Blueprint status casing
- **Root cause:** `blueprints/[id]/page.tsx` used `blueprintStatus.replace("_", " ")` for the inline status badge, rendering "deployed", "in review" (lowercase) instead of "Deployed", "In Review".
- **Fix:** Added `import { STATUS_LABELS }` and changed render to `STATUS_LABELS[blueprintStatus as keyof typeof STATUS_LABELS] ?? blueprintStatus.replace("_", " ")`.

### W-06 — Calendar page blank on API failure
- **Root cause:** `governor/calendar/page.tsx` had `.catch(() => setLoading(false))` with no error state. When API fails, `data` remains `null` and `{!loading && data && (...)}` renders nothing.
- **Fix:** Added `fetchError` state; show structured empty/error state with icon, message, and actionable context.

### W-07 — API Keys orphaned "Scopes" label
- **Root cause:** The scopes section was always rendered even when `validScopes` array is empty (e.g., API didn't return scopes), leaving an orphaned "Scopes" label.
- **Fix:** Wrapped the scopes section in `{validScopes.length > 0 && (...)}`.

### W-08 — Register form wizard shows all steps at once
- **Root cause:** The form had a P2-69 step progress indicator, but all fields were displayed simultaneously rather than step-by-step.
- **Fix:** Added `currentStep` state (0-2). Each step's fields are conditionally rendered. "Continue →" advances steps; "← Back" goes back; the submit button only appears on step 2. `stepValid[]` array gates the Continue button.

### M-01 — Login/Register labels low contrast on dark background
- **Root cause:** `FormField` renders labels with `text-text` (dark CSS variable), invisible on the dark authentication pages.
- **Fix:** Added `[&_label]:text-white/80` to the form wrapper in both login page and register form.

### M-02 — Login page missing "Create account" link
- **Fix:** Added "Create account →" link next to "Forgot your password?" in the login form footer. Also improved forgot-password link contrast from `/25` to `/40`.

### W-09 — Search palette only finds pages
- Already implemented in an earlier session (P2-584: agent + policy search). Verified correct. No change needed.

### W-10 / W-12 — Ask Intellios panel squishes content + intermittent failure
- **Root cause (W-12):** Trigger button `onClick={() => setOpen(true)}` could race with other click handlers in the sidebar. Event propagation could allow parent elements to close the panel immediately after open.
- **Root cause (W-10):** Backdrop had no background color, providing no visual context that the panel is an overlay.
- **Fix:** Added `e.stopPropagation()` to both trigger buttons. Changed backdrop from transparent `onClick` to `onMouseDown` with `bg-black/20` so it's a visible semi-transparent overlay.

### W-13 — Invite form no validation feedback
- **Fix:** Added explicit pre-submission validation in `InviteUserForm.handleSubmit` with clear inline error messages for missing email, invalid email format, and missing role.

### W-14 — Invite role dropdown missing "Designer" option
- **Fix:** Added `{ value: "designer", label: "Designer" }` to the `ROLES` array in `admin/users/page.tsx`. Designer is the legacy DB alias for Architect.

### W-15 — Viewer role exposes write-action lifecycle controls
- **Fix:** In `registry/[agentId]/page.tsx`, wrapped `<LifecycleControls>` in `{currentUser?.role !== "viewer" && (...)}` so viewers see the agent detail read-only without submit/approve/deploy buttons.

### M-04 — Admin Fleet "super-admin-only" error has no heading/structure
- **Fix:** Added `<Heading level={1}>Platform Fleet Overview</Heading>` with Globe icon, and a structured card explaining the restriction with a link to `/registry` for enterprise agents.

### M-08 — Governance Hub "Audit Trail →" button style inconsistency
- **Fix:** Standardized both `Audit Trail →` links to use the same `inline-flex items-center gap-1.5 rounded-lg border border-border bg-surface px-3 py-1.5 text-sm font-medium text-text-secondary` class.

### M-10 — "+ New Blueprint" button label confusing
- **Fix:** Changed label from "New Blueprint" to "Start Intake" to clarify that clicking the button opens an AI-guided intake session rather than a blank blueprint form.

### M-12 — "Enter to start" hint misleading (Mac-only ⌘ symbol)
- **Fix:** Changed `"⌘ Enter to start"` to `"↵ Enter to start"` in both `intake-quick-start.tsx` and `quick-start-modal.tsx`. Uses generic Enter arrow symbol that works across platforms.

### M-13 — Notification bell viewer/designer roles excluded
- The bell component was already fully wired (P2-606 implemented in Session 006). The issue was that viewer and designer roles were excluded from the `/api/notifications` `requireAuth` guard.
- **Fix:** Added `"designer"` and `"viewer"` to the `requireAuth` roles list in `src/app/api/notifications/route.ts`.
