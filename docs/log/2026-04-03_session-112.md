### P2-276 — Cross-version test comparison table in test harness
**File:** `src/components/registry/red-team-panel.tsx`

When red-team results are displayed and a previous version's run exists in localStorage history, a "Version Comparison" strip now appears between the summary header and the risk guidance banner. Shows both versions side by side: risk tier badge, score, and a delta like "↑ +2 attacks resisted · risk improved" in green or "↓ -1 attacks resisted · risk increased" in orange/red. Uses the existing `useRunHistory` data — zero additional API calls. ### P2-423 — Version selector in simulation sandbox
**Files:** `src/components/registry/simulate-panel.tsx`, `src/app/registry/[agentId]/page.tsx`

Added a `VersionEntry { id, version }` interface and an optional `allVersions` prop to `SimulatePanel`. When multiple versions exist, a version selector strip renders above the mode toggle (Chat / Red Team). Each version appears as a pill chip — the active version gets a violet ring, others are outline pills. Clicking a chip updates `activeVersionId` and `activeVersion` state and increments `chatKey` (clearing the chat on version switch so stale messages don't persist).

The registry page now passes `allVersions={versions.map((v) => ({ id: v.id, version: v.version }))}` to `SimulatePanel`. When an agent has only one version, the selector strip is hidden entirely.

**Brace balance:** simulate-panel 107/107 ✓ · registry page 671/671 ✓

### P2-584 — Extend command palette search to agents/blueprints/policies
**File:** `src/components/nav/command-palette.tsx`

Added `PolicyResult { id, name, type, description }` interface and `POLICY_TYPE_LABELS` record mapping type keys to human-readable labels (Safety, Compliance, Data Handling, Access Control, Audit). Added `policyResults` state and two refs: `allPoliciesRef` (the full policy list, fetched once on mount and held in memory) and `policiesFetchedRef` (prevents duplicate fetches). On mount, fetches `GET /api/governance/policies` and stores the result. On each query change, filters client-side by name/type/description — no debounce needed since the data is already local.

A "Policies" `Command.Group` renders after the "Agents" group, showing matching policies with a `ShieldCheck` icon, policy name, and type + description preview. Navigates to `/governance?policy={id}` on selection. Search placeholder updated to "Search pages, agents, and policies…".

**Brace balance:** 89/89 ✓

### P2-607 — Notification preferences routing matrix in Admin Settings
**Files:** `src/lib/settings/types.ts`, `src/app/admin/settings/page.tsx`

Extended `EnterpriseSettings.notifications` with two new optional fields:
- `routing`: a per-event, per-channel routing matrix covering 7 event types (blueprint_approved, blueprint_rejected, blueprint_deployed, policy_violation, sla_breach, review_assigned, anomaly_detected) × 3 channels (email, slack, pagerduty)
- `digestFrequency`: "immediate" | "daily" | "weekly" (default: "immediate")

Default values added to `DEFAULT_ENTERPRISE_SETTINGS.notifications`. Policy violation and SLA breach default to email+slack+pagerduty; critical anomaly similarly. Approval, deployment, and review events default to email only (with slack for deployed/rejected). Since enterprise settings are stored in a JSONB column and merged with defaults at read time (`deepMerge`), this requires zero schema migration.

The admin settings notifications section now has two new subsections inserted before the Slack channel config:

1. **Event Routing** — A compact bordered table with events as rows and Email/Slack/PagerDuty as columns. Each cell has a checkbox wired to the routing matrix. Alternating row backgrounds for readability. Descriptive note: "Channel must be configured above to receive alerts."

2. **Digest Frequency** — Three radio buttons (Immediate / Daily / Weekly) with contextual description text that updates based on the active selection. Weekly mode clarifies that critical alerts (SLA breach, anomaly) still fire immediately.
