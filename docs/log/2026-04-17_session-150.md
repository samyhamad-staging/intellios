## Decisions Made

- **Extend `blueprint-lifecycle.test.ts` rather than create a new file.** All top-level mocks for `db`, `schema`, `auth`, `publishEvent`, `parseBody`, `requestId`, `validator`, `enterpriseSettings`, and the route imports are already wired. Starting a new `review-governance.test.ts` would have meant duplicating ~110 lines of mock scaffold. Co-locating with the existing review describe block also puts the block-path and override-path tests next to each other for any future reviewer tracing the approval contract.
- **Four tests, not three.** The plan called for three (block, override, rollback). Adding the fourth — "reviewer with override flag still blocked" — costs ~15 lines and locks in the admin-only property of the override explicitly, separate from the blanket 409 case. Without it, a future refactor could accidentally let any role override as long as the flag is set.
- **Inline `makeBlockingReport()` helper in the describe block.** The existing `makeValidationReport` fixture predates ADR-019 and omits the `valid` field that `errorSeverityViolations` keys on. Extending the fixture would force every other test to reason about `valid` even though they don't care about it. Keeping the blocker shape local to the governance tests keeps the contract boundary small.
- **Leave the Zod `overrideReason ≥20 chars` refine uncovered at the unit-test layer.** `parseBody` is mocked in this suite — exercising Zod refines here would effectively test the mock. The refine is exercised end-to-end by parse-body integration tests and the route's Zod schema in review/route.ts is the source of truth. Noted the boundary rather than adding a ceremonial test.
- **Fix the `enterpriseRateLimit` mock gap as part of this session, not as a separate ticket.** The gap was blocking the entire `POST /api/blueprints (generate)` describe block on the full run. Skipping the fix would have meant leaving "pre-existing red" in a session whose whole purpose is locking in invariants with green regression tests. Six-line change, no scope creep concern.

## Deferred / Follow-up

- **Per-enterprise rate-limit unit tests (C4).** Session 148's `enterpriseRateLimit` implementation has no dedicated unit tests yet. The mock fix in this session unblocks the generate-route tests but does not exercise the sliding-window logic, Redis-failure fallback, or daily token budget reset. Candidate for a future session.
- **Encryption-key enforcement tests (C5).** Session 148's `SECRETS_ENCRYPTION_KEY` production-required check (ADR-018) is boot-time; worth a small test that simulates `process.env.NODE_ENV === "production"` and a missing key to assert the process exits rather than silently falling back.
- **`streamText` retry-budget test (C6).** Session 147's AI resilience layer has integration coverage via `resilient-generate.test.ts` but no test pinning the `maxRetries: 3` budget specifically.
- **Healthz route unit test.** Session 149 shipped `/api/healthz` without a test. Three cases worth locking: (1) all services up → 200 `status:"ok"`; (2) DB probe times out → 503 `db:"down"`; (3) Redis fallback → 200 `status:"ok"` with `redis:"fallback"` (fallback-is-not-degraded invariant from ADR-022).
