## Context

A validation check before Session 171 (demo video scaffolding) confirmed that the evidence package is already shipped as a JSON export with UI download buttons, an HTML report view, an S3 cache path, and two server-side routes. The session 168-resumed scaffolding of SCRUM-38/39/40/41/42 was based on the assumption that the evidence package itself was not yet shipped; that assumption was incorrect.

ADR-015 ("PDF Rendering of Evidence Package") is specifically about adding a *server-side programmatic PDF renderer* on top of the existing JSON export. The JSON export, the MRM assembly logic, the UI download button, the S3 cache, and the HTML report view are all shipped. ADR-015's scope — as stated in its Decision section — is exactly the PDF renderer at `GET /api/blueprints/[id]/evidence-package.pdf`.

This session corrects the memo, journal, and Jira to reflect that narrower scope. It does NOT close ADR-015 or promote it — ADR-015 remains `proposed` because the PDF renderer is not yet implemented. It does NOT start implementation. It does NOT create the memo's Confluence mirror (stays deferred).

#### SCRUM-40 — Proposal: RE-SCOPE (two options; Samy decides)

**Classification:** Partially shipped — Registry page proxies through blueprint ID. True registry-side route is absent.

**Current state:**
- `GET /api/registry/[agentId]/evidence` does NOT exist.
- The Registry detail page (`src/app/registry/[agentId]/page.tsx`) already provides evidence package download via two paths: (a) Exports dropdown linking to `/api/blueprints/${latest.id}/export/compliance` (line 715) and (b) `DownloadEvidenceButton` component in the MRM section (line 1275). Both use the blueprint ID (`latest.id`), not the agentId.
- Functional gap: a deployed agent accessed via its canonical agentId does not have a clean `/api/registry/{agentId}/evidence` URL. Consumers wanting to pull the evidence package for a deployed agent by agentId must first know the latest blueprint ID.

**Option A — RE-SCOPE (keep Story, clarify):** Retitle to "Add registry-path alias for evidence export: `GET /api/registry/[agentId]/evidence`." Scope: a thin route that looks up the latest `approved`/`deployed` blueprint for the given agentId, delegates to the blueprint-side evidence assembly, and returns the same JSON package with agentId-based audit event. Rationale: API cleanliness — external consumers (design partners, integrations) will call by agentId, not blueprintId. Priority: P2 (nice-to-have, not blocking demo or design partner).

**Option B — CLOSE as duplicate/unnecessary:** The registry page already provides download via blueprint-ID proxying. No current consumer calls this route. Close as Won't Do with a note that if a design partner requests an agentId-addressed API for evidence packages, it can be re-opened.

**Recommendation:** Option B (close). The registry UI already works. The route adds API surface without a concrete current consumer. It can be re-opened on design-partner pull.
