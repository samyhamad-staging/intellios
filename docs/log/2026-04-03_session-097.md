## Key Decisions

- **`ruleId` as acknowledgement key, not array index** — `Violation.ruleId` is stable across re-validation runs. Using index would cause acknowledgements to "jump" to wrong violations if the list order changes. `ruleId` is the correct unique identifier.
- **Acknowledgements are session-scoped (not persisted)** — Persisting acknowledgements would require a new DB column or table. Session-scoped is correct for the current stage: the architect performs review in one sitting. A future enhancement could persist acknowledgements to an audit log.
- **Warnings require acknowledgement before submission** — This creates a governance paper trail that the architect consciously reviewed each advisory violation. Without this gate, architects could submit with unread warnings, which undermines the governance layer's value for compliance officers reviewing the queue.
- **Errors remain unchanged** — Errors still hard-block submission. The acknowledgement workflow is additive on top of existing error blocking.
- **Suggestions collapse when acknowledged** — Once acknowledged, the suggestion box is hidden. This reduces visual noise and signals "this is resolved (accepted)" clearly.
- **Clean governance section condition narrowed** — `validationReport?.valid` was previously showing the clean "Governance" section even when warnings existed (since `valid = true` means no errors, but warnings can still be present). Fixed to `violations.length === 0` so warnings-only states correctly show the violations/acknowledgement section instead.
