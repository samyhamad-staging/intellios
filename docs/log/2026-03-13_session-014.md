## Actions

### Phase 11: Policy Lifecycle Audit

#### 1. `src/lib/events/types.ts` ‚Äî Modified
Added `"policy.updated"` and `"policy.deleted"` to the `EventType` union. These mirror `AuditAction` 1:1 per the existing convention.

#### 2. `src/lib/audit/log.ts` ‚Äî Modified
Added `"policy.updated"` and `"policy.deleted"` to the `AuditAction` union.

#### 3. `src/app/api/governance/policies/route.ts` ‚Äî Modified
Added `import { writeAuditLog }` and a `void writeAuditLog(...)` call after successful policy creation:
- `action: "policy.created"`, `toState: { name, type, ruleCount }`
- Previously, `policy.created` was defined in `AuditAction` but never called from the POST handler ‚Äî this was a silent audit gap.

#### 4. `src/app/api/governance/policies/[id]/route.ts` ‚Äî Modified
Added `import { writeAuditLog }` and audit calls in both PATCH and DELETE:
- PATCH: `action: "policy.updated"`, `fromState: { name, type, ruleCount }` (pre-update), `toState: { name, type, ruleCount }` (post-update)
- DELETE: `action: "policy.deleted"`, `fromState: { name, type, ruleCount }` (final state before deletion)
- Both use fire-and-forget (`void`) consistent with the rest of the audit infrastructure ‚Äî audit failure never blocks the primary operation.

#### 5. `src/app/audit/page.tsx` ‚Äî Modified
Extended `ACTION_LABELS` and `ACTION_COLORS` to include all remaining action types:
- Added `policy.updated` (orange), `policy.deleted` (rose/dark), `blueprint.report_exported` (teal), `intake.contribution_submitted` (sky)
- All 10 `AuditAction` values now have explicit display labels and color tokens

### Phase 12: Admin User Management

#### 6. `src/app/api/admin/users/route.ts` ‚Äî NEW
- **GET** ‚Äî list all users in the admin's enterprise; `admin` only; platform admins (no `enterpriseId`) see all users; returns id/name/email/role/enterpriseId/createdAt (no passwordHash)
- **POST** ‚Äî create a new user; email uniqueness check; bcrypt hash at cost 12; scopes new user to `authSession.user.enterpriseId`; returns 201 with user record; conflicts return 409

#### 7. `src/app/api/admin/users/[id]/route.ts` ‚Äî NEW
- **PATCH** ‚Äî update user `name` and/or `role`; `admin` only; enterprise-scoped access guard; self-protection: admin cannot change their own role away from `admin`

#### 8. `src/app/api/me/route.ts` ‚Äî Modified
Added `id` to the response (both at the top level and inside the `user` object). The admin users page needs the current user's ID to enforce the "cannot edit own role" UX protection.

#### 9. `src/app/admin/users/page.tsx` ‚Äî NEW
Client component with three sections:
- **Role summary strip**: 4-stat grid showing user counts per role (Designer/Reviewer/Compliance Officer/Admin)
- **Create User form**: collapsible blue panel with name, email, role select, and temporary-password fields; calls POST `/api/admin/users`; on success, adds new row to the table
- **User table**: sorted alphabetically; columns: User (avatar initial + name), Email, Role (inline editable), Joined date
  - `InlineRoleEditor`: each role cell shows badge + "Edit" link; clicking opens an inline `<select>` + Save/Cancel; calls PATCH `/api/admin/users/[id]`; own row shows "(you)" with no edit affordance
  - Self-protection in the API: admin cannot change their own role via PATCH

#### 10. `src/app/layout.tsx` ‚Äî Modified
Added "Users" nav link for `admin` role only, after Dashboard. Positioned at the far right of the admin's nav cluster.
