# Session 156 — 2026-04-18

## Context

Session 155 closed the PRR arc (H3→H5→H6→H4) by shipping the ADR-026 webhook retry + DLQ + replay infrastructure. That session's own deferred list opened with a single entry: *"Admin UI 'DLQ count' badge on the webhooks list + a DLQ-filtered delivery view. The backend supports it today; the UI wire-up is a cosmetic session."* Session 156 is that cosmetic session plus the replay button wiring — the product-facing UX that turns the ADR-026 plumbing into a thing operators can actually use.

Session 155's response shape was already usable from curl, but three gaps remained between "the feature exists" and "an admin can operate it":

1. The webhooks list page (`/admin/webhooks`) did not know `dlqCount` exists — no badge, no sort signal, no way to see which webhook is carrying unactioned dead-letter entries.
2. The delivery log under each webhook card rendered `pending | success | failed` rows with no way to see just the DLQ, no `errorClass` column, and no column distinction for pending-with-scheduled-retry vs failed.
3. There was no Replay button anywhere in the UI. ADR-026 shipped a replay *endpoint*; invoking it required hitting the API directly, which is the wrong ergonomic for "this one delivery looks stuck, poke it."

Scope this session: close those three gaps in the existing admin webhooks page, extend the backend list and detail endpoints with the new response fields, and add the two backend test files that were missing.

## Decisions

1. **DLQ chip is a linked action, not two clicks.** The obvious UX shape would have been two independent affordances — a DLQ count chip on the card header showing `N in DLQ`, and the filter tabs inside the expanded card. The admin who sees `3 in DLQ` on the card header wants to see those three specific rows immediately, not click the card open, scan for the filter, and click DLQ. So the DLQ chip is wired to `openDlqView()` which both expands the card and applies the `dlq` filter in one action. The filter tabs still exist inside the expanded card for the admin who arrived via a different path and wants to switch filters after the card is already open. The chip is the entry point; the tabs are the switcher. Two affordances doing different jobs, not one affordance split in two.
2. **Replay button renders on `dlq | failed | success`, not just `dlq`.** The session 155 ADR explicitly supports replaying successful deliveries (the use case: "I need to re-fire this event because the downstream system's first processing was wrong"). Restricting the button to `dlq` would have been UI-level correctness-theater that forced operators back to curl for a legitimate use case. Rendering the button everywhere it's semantically valid is what makes it a tool rather than a recovery-only recovery tool. The `pending` state correctly does not render the button — a pending delivery is about to fire on its own; a replay while pending would reset the attempt counter and re-queue, which is distinct from "the system has already given up and I want to try again" and would confuse the audit trail.
3. **Pending rows show `nextAttemptAt`, not `lastAttemptedAt`.** The "When" column's semantics shift based on row state. For pending rows, the operationally relevant fact is "when will this fire next" — `in 2h`, `in 35m`. For non-pending rows, the relevant fact is "when was the last attempt" — `3m ago`, `2h ago`. Using a single column label ("When") that switches semantics on row state would be bad UX; using two separate columns doubles the table width for a signal that's only non-null on one row type. The compromise: keep the "When" label and render a relative-time string that reads sensibly either way — `in 2h` for pending and `2h ago` for the rest — with `formatInterval` handling the future-direction case and the existing formatter handling the past-direction case. A reader scanning the column gets the semantics from the direction word without needing to read the column header.
4. **Correlated subquery over JOIN+GROUP BY for `dlqCount`.** A join+group-by over `webhooks × webhook_deliveries` would produce the same numbers and would be the pattern that most ORMs' query-builder tutorials suggest for this shape. The correlated subquery is uglier to look at and wins on two counts: (a) the result row count stays identical to `webhooks` (no risk of a GROUP BY misstep accidentally collapsing or duplicating rows), and (b) ordering semantics are trivial — `ORDER BY webhooks.createdAt DESC` works without any compound-key gymnastics. The subquery is a one-time computed column, not a repeated table scan at scale (`dlq` is expected to be small, and the partial index from migration 0040 covers `status='pending'`; a functional index on `status='dlq'` is a future addition if this query ever shows up in slow-query logs).
5. **DLQ chip renders only when count > 0.** An "ever-present" chip (e.g., rendered grey when count=0, red when count>0) would produce visual noise that most admins see most of the time on most webhooks. The absence of the chip is a signal — if the card has no red chip, the DLQ is empty. This is a slightly lossy signal (it collapses "zero" and "haven't loaded yet" into the same visual state), but the admin's mental model is "if something is wrong, it would show up here," which is served by the absence-is-good contract. The signal that matters is *presence* of red, not *absence* of grey.
6. **Filter tab DLQ count is redundant with the card chip — intentionally.** The filter tabs row renders `DLQ (3)` not just `DLQ`. That count is the same number that appears on the card header chip. The redundancy exists because the filter tabs are the per-card switcher and the chip is the list-level summary; a user who opened the card via a direct link (e.g., from a Slack alert) might not have seen the chip. Both surfaces render the same underlying count because the alternative — rendering it only on one — would leave one reading path under-informed.
7. **`?status=` unknown-value fallback is silent, not 400.** An admin hitting `?status=bogus` is almost certainly arriving via a stale URL (browser back button, shared link that outlived a UI change). A 400 would break a working page for a cosmetic query-param issue. The fallback is "ignore the filter, return the default view" — the URL is wrong but the page still works. For body schemas this would be incorrect (body validation is part of the contract); for URL filter params it's the right trade-off.

## Deferred / follow-ups

- Test-delivery DLQ presentation — per session 155's decision #5, a failed `deliverWebhookTest` call lands in `dlq` immediately. The admin UI should label those rows visually distinctly (e.g., a "Test" tag next to the status badge) so the DLQ view doesn't conflate diagnostic runs with production failures. The DB already supports the distinction; the UI doesn't surface it yet.
- DLQ growth-rate alerting — e.g., `> 10 new DLQ entries per enterprise per hour`. Belongs in the alerts pipeline, not the admin UI. Session 155 noted this; still pending.
- `POST /api/admin/webhooks/deliveries/[id]/cancel` — force a pending delivery to DLQ without replay. Surfaced when an admin knows an event is no longer relevant (user deleted, entity hard-reset). One-session addition; no UI wiring yet.
- Per-webhook retry-schedule override — the `max_attempts` column is already per-row (see ADR-026 Decision #4), but there's no admin UI to set it. Currently requires a direct DB write. Session-sized form addition to the webhook detail view.
- CSV export performance — current implementation is client-side and materializes the full current filter result. For enterprises with >10k deliveries per webhook this will get sluggish. Future: server-side streaming export (`?format=csv`).
- Delivery payload body capture — session 155's deferred list included this (rolling window of response bodies per delivery). Still pending. Requires a new table or a JSONB column; session-sized.

## Summary

The admin webhooks page now surfaces ADR-026's retry-and-DLQ machinery as an operator tool rather than a set of curl endpoints. A webhook card shows a red `N in DLQ` chip when DLQ entries exist; clicking it expands the card with the DLQ filter already applied. The delivery log renders `dlq` rows in red, splits "When" between `nextAttemptAt` (pending) and `lastAttemptedAt` (everything else), surfaces `errorClass` as a subtitle, and renders a Replay button on `dlq | failed | success` rows. Replay POSTs to the session-155 endpoint and handles the 422 inactive-webhook case with a dedicated toast message. Filter tabs (All / Pending / DLQ-with-count) let operators narrow the table without leaving the card; CSV export respects the active filter and includes the new `errorClass` / `nextAttemptAt` columns.

Backend: `GET /api/admin/webhooks` now returns `dlqCount` per row via correlated subquery. `GET /api/admin/webhooks/[id]` accepts `?status=pending|success|failed|dlq` with unknown-value fallback; `?status=dlq` widens the delivery limit from 20 to 50; `errorClass` and `nextAttemptAt` join the delivery select. Two new test files (5 + 9 cases) cover the list aggregation, detail filter, limit widening, and where-clause shape.

Full suite **725/725 green** (+14 from 711), 32 files (+2 from 30). Typecheck unchanged — 7 pre-existing errors in `blueprint-lifecycle.test.ts` remain; zero new. No new ADR; this session operationalizes ADR-026 without adding new semantic decisions.
