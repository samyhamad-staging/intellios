## OQ-009 — Evidence-package PDF renderer stack (resolution proposed)

### The decision

Three candidate stacks per ADR-015 + OQ-009:

1. **Node-native PDF library** (`pdf-lib` or `pdfkit`) — full programmatic control, deterministic byte-identical output, ~200KB bundle impact, but the Python reference is 1,769 lines and reimplementing layout in TypeScript is multi-session work.
2. **Headless Chromium via Playwright** — reuse the existing 1,230-line HTML report shell at `src/app/blueprints/[id]/report/page.tsx` (already heavy with `print:` Tailwind variants), print to PDF in a headless browser. Lower implementation cost; large deploy footprint without Vercel-friendly Chromium build.
3. **Puppeteer** — same trade-offs as Playwright, marginally lighter dependency surface.

### Recommendation: **Option 2 (Playwright + `@sparticuz/chromium-min`)**

Three reasons.

**Reuse of an asset that already exists.** The HTML report at `src/app/blueprints/[id]/report/page.tsx` is 1,230 lines of carefully-laid-out React with Tailwind print variants integrated throughout (`print:break-before-page`, `print:hidden`, `print:max-w-none`, etc.). It is, today, the canonical on-screen rendering of every MRM section. Headless Chromium prints this directly. pdf-lib forces a parallel implementation that drifts from this canonical view as `MRMReport` evolves — exactly the "two rendering paths must stay in sync" risk ADR-015 calls out.

**Vercel deployability is solved.** Standard Chromium binary is ~280MB compressed (over Vercel's 250MB function limit). `@sparticuz/chromium-min` is the standard 2026 pattern: ~50MB binary, hosted-elsewhere on first cold start, designed specifically for serverless PDF rendering on Vercel/AWS Lambda. Combined with Playwright's `chromium.launch({ executablePath })`, the bundle stays under limit. Cold-start latency (~2-4s) is acceptable because the result is cached to S3 and subsequent reads serve via signed URL redirect (same pattern as the existing JSON export).

**Determinism risk is mitigated.** ADR-015's concern is byte-identical rendering for "signed/archived distribution." Headless Chromium across versions does drift. Mitigation: the S3-cached PDF (`evidence/{id}/{version}.pdf`) is a write-once artifact — once generated, that exact byte sequence is what's served. The "deterministic across environments" property matters when re-rendering; the cache means we never re-render. If a future ADR adds cryptographic signing, the cache-on-write approach is what makes signing meaningful.

### Cost of the recommendation

- Two new dependencies: `playwright-core` (~6MB, browser-launch APIs only — not full Playwright) and `@sparticuz/chromium-min` (~150KB code, ~50MB Chromium binary loaded at runtime from a hosted URL).
- One new env var: `CHROMIUM_REMOTE_EXECUTABLE_URL` for `@sparticuz/chromium-min` to fetch the binary on cold start.
- Function memory: bump from default 1024MB to 2048MB on the new route only.
- Cold-start: 2-4s on first request per warm function instance; ~50-100ms on warm.

### What's deferred

- Cryptographic signing of the PDF (ADR-015 explicitly out of scope).
- White-label per-tenant theming (H3).
- Switching to Option 1 (`pdf-lib`) is reversible: the route's S3-cache-and-serve logic + auth/audit logic is renderer-agnostic. Only the `renderEvidencePDF()` function would need to swap.

OQ-009 currently sits as `proposed` in this session log. Before I add `playwright-core` and `@sparticuz/chromium-min` to `src/package.json`, I want explicit confirmation. The alternatives — pdf-lib (high upfront cost, low ongoing risk) or deferring this decision — are real options.

### Architectural notes

- **Package shape parallel, not shared.** The PDF route inlines the same package assembly as the JSON route (`assembleMRMReport()` + approval chain + quality score + test runs). A shared helper (`assembleEvidencePackage()`) is a reasonable future refactor but was held off to avoid scope creep on the JSON route. Drift risk is low — both routes call `assembleMRMReport()` for the bulk of the data.
- **Determinism strategy.** S3 cache (`evidence/{id}/{version}.pdf`) is write-once: the first export of a given blueprint version renders the PDF; every subsequent export serves the cached bytes. Chromium-version drift across renders is therefore a non-issue.
- **Cold-start.** First request per warm function instance: ~2-4s (Chromium binary download + launch). Warm requests: ~50-200ms.
- **Function memory.** The new route should run with at least 2048MB. A `vercel.json` per-route override is a follow-up; default Vercel function memory may be sufficient for short MRMs but headroom matters for long audit chains. Captured as a TODO in the route file's docstring.

### Lesson — sandbox validation gap

Two production type errors past the sandbox `esbuild` validation. Both were real bugs:

- The first (events payload union) was a discriminated-union violation Vercel's `tsc` caught and esbuild bundling did not.
- The second (chromium headless union) was a third-party library type that bled into the local contract — `@sparticuz/chromium-min` types `headless` more loosely than Playwright accepts.

Pattern captured in memory: `intellios_sandbox_validation_gap.md`. Future src/ pushes touching shared type contracts should budget for at least one fix-up cycle. The audit-trail discipline is not "esbuild bundle ran" — it is "code green on Vercel" + "implementing commit on `main`."
