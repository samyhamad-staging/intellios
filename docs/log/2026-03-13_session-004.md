# Session 004 — 2026-03-13

## Objective

Implement application-level multi-tenancy enforcement as the final P0 Post-MVP Phase 1 item. Every API route must be scoped so that a user at Enterprise A cannot read or write Enterprise B's data.

## Context at Start

All Post-MVP Phase 1 hardening was complete (auth, RBAC, audit log, rate limiting, Zod validation, security headers, request correlation IDs, env var validation, Intake Engine UX). Multi-tenancy was the last P0 gap — schema columns existed (`enterprise_id` on `intake_sessions` and `governance_policies`) but were not populated or enforced anywhere. The `users`, `agent_blueprints`, and `audit_log` tables had no `enterprise_id` column at all.

| `POST /api/governance/policies` | Sets `enterpriseId` from auth session (not request body) |

## Key Design Decisions

- **`admin` role has cross-enterprise access** — necessary for platform operations, support, and compliance oversight across tenants.
- **`enterpriseId` is denormalized to `agent_blueprints`** — avoids a JOIN through `intake_sessions` on every blueprint read. Set at creation, immutable.
- **`enterpriseId` on `audit_log`** — enables enterprise-scoped compliance officer queries without joining other tables.
- **`assertEnterpriseAccess()` returns null (allow) or 403 NextResponse** — clean function signature; caller returns immediately on non-null.
- **Governance policies GET returns global + enterprise** — global policies (null enterpriseId) apply to all enterprises; enterprise-specific policies overlay them.
- **Governance policies POST sets enterpriseId from auth, not body** — closing an injection vector where a caller could create policies for a different enterprise.

## Summary

Multi-tenancy is fully enforced at the application layer across all routes. The only remaining Phase 1 items are P2 (distributed rate limiting with Redis, deployment pipeline). Phase 1 is functionally complete for enterprise use.
