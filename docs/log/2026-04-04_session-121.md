### W3-03: Policy Binding — Per-Agent Scope Selector

**Investigation:**
- Read `src/lib/db/schema.ts` — `governancePolicies` table has `enterpriseId` (tenant scope) but no per-agent scope. All policies apply to all blueprints in the enterprise with no per-agent override.
- Read `src/lib/governance/validator.ts`, `load-policies.ts`, `types.ts` — validator calls `loadPolicies(enterpriseId)`, which returns all enterprise policies. No `agentId` parameter anywhere in the chain.
- Read `src/app/api/governance/policies/route.ts` and `[id]/route.ts` — Zod schemas had no `scopedAgentIds` field. PATCH creates a new version row (Phase 22 versioning pattern).
- Read `src/components/governance/policy-form.tsx` — form has 1050+ lines with Policy Details, Rules, Impact Simulation sections. No scope field.
- Read parent pages `policies/new/page.tsx` and `policies/[id]/edit/page.tsx` — minimal, pass form values to API.

**Implementation:**

1. **Migration** (`src/lib/db/migrations/0035_policy_scope.sql`):
   ```sql
   ALTER TABLE governance_policies
     ADD COLUMN IF NOT EXISTS scoped_agent_ids JSONB DEFAULT NULL;
   ```

2. **Schema** (`src/lib/db/schema.ts`): Added `scopedAgentIds: jsonb("scoped_agent_ids").default(null)` with comment explaining semantics.

3. **Types** (`src/lib/governance/types.ts`): Added `scopedAgentIds: string[] | null` to `GovernancePolicy` interface with JSDoc.

4. **`load-policies.ts`**: Added optional `agentId?: string | null` parameter. Post-query JS filter: if `agentId` provided, only return policies where `scopedAgentIds IS NULL` or `scopedAgentIds.includes(agentId)`. Simple in-memory filter avoids complex JSONB SQL.

5. **`validator.ts`**: Added optional `agentId?: string | null` 4th parameter. Passed to `loadPolicies(enterpriseId, agentId)` when policies are not pre-supplied.

6. **API `policies/route.ts`** (POST): Added `ScopedAgentIdsSchema = z.array(z.string().uuid()).nullable().optional()` to `CreatePolicyBody`; inserts `scopedAgentIds: body.scopedAgentIds ?? null`.

7. **API `policies/[id]/route.ts`** (PATCH): Added `scopedAgentIds` to `UpdatePolicyBody`; computes `newScopedAgentIds` from body (if key present) or inherits from previous version; inserts into new version row.

8. **`policy-form.tsx`** (major):
   - Added `Search` from lucide-react, `Checkbox` from `@/components/catalyst/checkbox`
   - Added `AgentRef { agentId, name }` interface
   - Extended `PolicyFormValues` with `scopedAgentIds: string[] | null`
   - New state: `scopedAgentIds`, `agentSearch`, `availableAgents`, `agentsLoading`
   - `useEffect` fetches `/api/registry` on mount (non-readOnly); populates `availableAgents`
   - `filteredAgents` memoized by `agentSearch`
   - `toggleAgent(agentId)` toggles an agent in/out of `scopedAgentIds`
   - Draft save/restore extended to persist `scopedAgentIds`
   - `handleSubmit` now passes `scopedAgentIds`
   - **New "Agent Scope" section** inserted between Policy Details and Rules:
     - Radio toggle: "All agents (enterprise-wide default)" | "Specific agents only"
     - When "Specific agents": search input, scrollable agent list with Catalyst Checkbox, selection summary
     - readOnly mode: shows current scope as badge pills (resolves names from `availableAgents`)

9. **`policies/[id]/edit/page.tsx`**: Added `scopedAgentIds: string[] | null` to `Policy` interface; included in `initialValues`.

10. **`governance/page.tsx`**: Added `scopedAgentIds: string[] | null` to `Policy` interface; added amber scope badge in policy list: `{scopedAgentIds.length} agent(s)` with tooltip showing count.

11. **Validate route** (`api/blueprints/[id]/validate/route.ts`): Wired `blueprint.agentId` as 4th param to `validateBlueprint`.

### W3-04: Catalyst Alert Adoption

**Investigation:**
- Read `src/components/catalyst/alert.tsx` — existing Catalyst `Alert` is a Headless UI Dialog-based alert modal (not an inline banner).
- The tracker intent was "Replace div-based info/warning/error patterns with Catalyst Alert (info/warning/error/success variants)" — the existing component doesn't have these variants.
- Decision: extend `catalyst/alert.tsx` with a new `InlineAlert` exported component.

**Implementation:**

1. **Extended `catalyst/alert.tsx`** with `InlineAlert`:
   ```tsx
   const inlineVariants = {
     error:   'border-red-200 bg-red-50 text-red-700',
     warning: 'border-amber-200 bg-amber-50 text-amber-800',
     info:    'border-blue-200 bg-blue-50 text-blue-700',
     success: 'border-emerald-200 bg-emerald-50 text-emerald-700',
   }
   export function InlineAlert({ variant = 'info', className, children }) { ... }
   ```
   - Adds `role="alert"` for accessibility
   - 4 variants with matching border/background/text color

2. **Adopted in `policies/new/page.tsx`**: replaced ad-hoc error div with `<InlineAlert variant="error">`.

3. **Adopted in `policies/[id]/edit/page.tsx`**: replaced error div (`variant="error"`) and platform policy warning div (`variant="warning"`).

4. **Adopted in `blueprints/page.tsx`**: replaced the `AlertCircle` + error div pattern with `<InlineAlert variant="error" className="flex items-center gap-2">`.
