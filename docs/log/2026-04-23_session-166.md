## Infrastructure Setup (pre-walkthrough)

### Problem 1 — PostgreSQL port conflict
Native Windows postgres service (PID 6856) held port 5432. Docker could not bind.

**Fix:** Changed `docker-compose.yml` `ports` from `"5432:5432"` to `"5433:5432"`. Updated `DATABASE_URL` in `.env.local` to `localhost:5433`.

### Problem 2 — Missing `geist-mono` font files
`geist` npm package v1.7.0 includes only TTF files; no `GeistMono-Variable.woff2`. `localFont` in `layout.tsx` threw at startup.

**Fix:** Removed both `localFont` declarations from `src/app/layout.tsx`. Kept only `Plus_Jakarta_Sans`.

### Problem 3 — Turbopack `@react-aria/interactions` resolution failure
`@headlessui/react` v2.2.10 (upgraded from v2.2.9 during a `npm install geist` run) adds `@react-aria/interactions` as a dependency. Turbopack's `resolveAlias` in `next.config.ts` pointed to the correct file but used Windows backslash paths from `path.resolve()` — Turbopack on Windows rejects backslash-separated paths.

**Root cause path:** `app/login/page.tsx` → `catalyst/button.tsx` → `@headlessui/react` → `@react-aria/interactions` → alias resolves to backslash path → Turbopack 500.

**Fix (session 166 continuation):** Updated `src/next.config.ts` `resolveAlias` values to append `.replace(/\\/g, "/")` to all four `@react-aria/*` alias paths. Server restart required for Turbopack config changes; resolved all 500 errors.

**Intermediate workaround (earlier in session):** Removed Catalyst `Heading` import from `src/app/error.tsx` and `src/app/not-found.tsx` (replaced with plain `<h1>`). This fixed those two pages but not the login page.

### Problem 4 — Dev server couldn't start from WSL bash
Windows `.cmd` binaries in `node_modules/.bin/` don't execute from WSL. `npm run dev` must run from Windows PowerShell.

**Fix:** Samy ran `npm install` and `npm run dev` from PowerShell.

### Problem 5 — `ANTHROPIC_API_KEY=""` override
An empty `ANTHROPIC_API_KEY` in the WSL bash environment overrode `.env.local`. Next.js gives system env vars precedence.

**Fix:** Unset the variable in bash; server started from PowerShell where the empty var wasn't set.

## Bugs Found and Fixed During Walkthrough

### Bug 1 — `translate.ts`: Agent name with spaces rejected by Bedrock
**Symptom:** `CreateAgent` failed with a Bedrock validation error. Agent name `"Retail Bank Customer Service Agent"` contains spaces; Bedrock pattern is `([0-9a-zA-Z][_-]?){1,100}`.

**Fix (src/lib/agentcore/translate.ts:138):**
```ts
// Before
const agentName = truncate(rawName.replace(/[^0-9a-zA-Z\-_ ]/g, "_"), 100);

// After
const rawName = abp.identity.name || "Unnamed-Agent";
const agentName = truncate(
  rawName.replace(/\s+/g, "-").replace(/[^0-9a-zA-Z_-]/g, "_").replace(/[-_]{2,}/g, "-"),
  100
);
```

### Bug 2 — `deploy.ts`: `CreateAgentActionGroup` called while agent in `CREATING` state
**Symptom:** `CreateAgentActionGroup` returned a Bedrock error. Bedrock requires agent to be in `NOT_PREPARED` state before action groups can be added.

**Fix (src/lib/agentcore/deploy.ts:145):** Added a polling loop between Step 1 (CreateAgent) and Step 2 (CreateAgentActionGroup):
```ts
// Step 1b: Wait for agent to leave CREATING state
for (let attempt = 0; attempt < 40; attempt++) {
  await sleep(500);
  try {
    const getResult = await client.send(new GetAgentCommand({ agentId }));
    if (getResult.agent?.agentStatus !== "CREATING") break;
  } catch { /* Transient — keep waiting */ }
}
```

### Bug 3 — SCRUM-27: Default model `anthropic.claude-3-5-sonnet-20241022-v2:0` unavailable in us-east-1 for InvokeAgent
**Symptom:** InvokeAgent returned "The ARN you specified was not found."

**Resolution:** Switched enterprise settings `foundationModel` to cross-region inference profile `us.anthropic.claude-3-5-sonnet-20241022-v2:0`. Deprecated Blueprint 1 (deployed with old model), deployed Blueprint 2 with corrected model.

- Response: `AGENTCORE_INVOKE_FAILED: InvokeAgent failed: Access denied when calling Bedrock.`
- **Root cause:** The server's IAM credentials (from `~/.aws/credentials`) have `bedrock-agent:*` permissions (sufficient for CreateAgent) but lack `bedrock-agent-runtime:InvokeAgent` (required for live invocations). These are separate IAM actions on separate service namespaces.
- **Finding:** The invoke endpoint is wired correctly — Bedrock responds with a proper AccessDeniedException rather than a network or routing error. The gap is purely IAM policy: needs `bedrock-agent-runtime:InvokeAgent` on `arn:aws:bedrock:us-east-1:540628117221:agent/SBVVOLDT3S`.
- **Next step:** SCRUM-27 scope update — add `bedrock-agent-runtime:InvokeAgent` to the IAM user policy; re-test.
