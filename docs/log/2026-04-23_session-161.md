### Part B — SCRUM-8 (ADR-018) — first dogfood of ADR-029

7. **Recon: ADR-018 implementation already landed.** Commit `9de4f8f` (session 148, 2026-04-17) shipped the full ADR-018 implementation as part of the P0 hardening bundle:
   - `src/lib/env.ts` — Zod schema conditionally requires `SECRETS_ENCRYPTION_KEY` (64-hex regex) when `NODE_ENV=production`; optional elsewhere. `NODE_ENV` enum with `development` default.
   - `src/lib/crypto/encrypt.ts` — `getKey()` adds defensive runtime throw if `NODE_ENV=production` and key is missing/malformed (unreachable under normal boot; covers the test-stubbing edge case ADR-018 called out).
   - `src/.env.example` — documents the key under "Encryption of secrets at rest (required in production, ADR-018)" with the `randomBytes(32).toString('hex')` generation command.
   ADR-018 itself was never promoted from proposed → accepted; the Jira Story (SCRUM-8) never closed. The session-148 commit message names this C5 but no surface was updated to match.
8. **Gap: no test asserts env.ts rejects prod boot without the key.** `src/__tests__/security/crypto-encrypt.test.ts` covers the crypto module's runtime behavior but does not exercise the Zod schema's production-boot gate. Added a test module that mutates `NODE_ENV` + `SECRETS_ENCRYPTION_KEY` via `vi.stubEnv` and re-imports `@/lib/env` to assert (a) boot fails without the key in production, (b) boot fails on malformed keys, (c) boot succeeds with a valid key, (d) dev/test boots without the key.
9. **SCRUM-8 transitioned to In Progress** — first ADR-029-compliant transition recorded against an implementation Story.
10. **ADR-018 flipped to accepted** across all four surfaces per ADR-029 discipline: repo file, repo `_index.md`, Confluence ADR page, Confluence ADR catalog.
11. **SCRUM-8 closed** — comment citing commit `9de4f8f` (implementation) + this session's test commit + ADR-018 acceptance; transitioned to Done.
