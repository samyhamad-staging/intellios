## Summary

**Stakeholder Requirement Lanes** — closes the gap between knowing that domain specialists were
consulted and capturing what they actually said.

### The Problem
`stakeholdersConsulted` in Phase 1 was a boolean proxy. It recorded participation but not content.
A compliance officer's FINRA Rule 3110 requirements would only appear in the blueprint if the
designer knew to raise them during the Phase 2 conversation. In Fortune 500 financial services,
with separate compliance, risk, legal, security, IT, operations, and business stakeholders, this
was a systematic completeness gap.

### ContributionDomain + StakeholderContribution
New `ContributionDomain` type: 7 values (`compliance`, `risk`, `legal`, `security`, `it`,
`operations`, `business`). New `StakeholderContribution` interface: id, sessionId, domain,
contributorName, contributorRole, fields (domain-keyed JSONB), submittedAt.

### DB: `intake_contributions` table
Migration `0007_intake_contributions.sql`. New Drizzle table added to schema. Applied.

### API: POST + GET /contributions
`POST /api/intake/sessions/[id]/contributions` — Zod validation on domain enum + required fields.
Enterprise access guard. Writes `intake.contribution_submitted` audit entry.
`GET /api/intake/sessions/[id]/contributions` — List all contributions for session.

### Phase 2: Domain-Adaptive Form + Sidebar Panel
`StakeholderContributionForm` — domain select reveals domain-specific fields. Separate fields for
each of the 7 domains (3 fields each). Submit calls POST /contributions.
`StakeholderContributionsPanel` — shown at bottom of Phase 2 sidebar inside `IntakeProgress`.
Count badge, per-contribution cards, Add Contribution affordance.

### System Prompt Injection
`buildContributionsBlock()` generates a `## Stakeholder Requirements` section with per-domain
attributed headers and field values. Claude instructed to incorporate verbatim.
`buildIntakeSystemPrompt(payload, context?, contributions?)` — contributions injected between
Enterprise Context block and Current State block.

### Phase 3 Review Screen
`IntakeReview` shows contributions panel before section cards — full field content with attribution.
Reviewing designer sees exactly what each stakeholder submitted before generating.

### MRM Report Section 11
`MRMReport.stakeholderContributions` — attributed contribution array. 6th DB query in
`assembleMRMReport()`. Evidence grade: every stakeholder requirement that went into the blueprint
is named, attributed, and timestamped in the compliance report.
