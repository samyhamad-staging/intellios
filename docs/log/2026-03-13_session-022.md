# Session 022 — 2026-03-13

## Summary

Phase 19: Deployment Health & Governance Drift Detection. Adds continuous governance posture monitoring for deployed agents. Once an agent is marked `deployed`, any enterprise policy change could silently cause it to drift out of compliance. Phase 19 closes this gap with a deterministic (no-AI) health check engine that re-evaluates deployed blueprints against the current policy set on every policy mutation, records health status per logical agent, notifies compliance officers on `clean→critical` and `critical→clean` transitions only, and exposes a new `/monitor` page with per-agent health badges and manual re-check controls.

Key design choices:
- `evaluatePolicies()` (pure rule engine) used for health checks — never `validateBlueprint()` — so checks are cheap enough to fire on every policy save with zero AI cost.
- UPSERT pattern on `deploymentHealth.agentId` — one row per logical agent, updated on every check; historical health traceable via audit log.
- Transition-based notifications only — prevents alert fatigue from repeated same-status checks.
- Fire-and-forget initial check on deployment — does not block the PATCH /status response.
- Circular import (`policy-impact-handler.ts` ↔ `audit/log.ts`) safe via Node.js module-cache singleton — identical to the existing `notifications/handler.ts` pattern.

### 5. `src/lib/monitoring/health.ts` — Created

Exports:
- `HealthStatus` type (`"clean" | "critical" | "unknown"`)
- `HealthCheckResult` interface
- `checkDeploymentHealth(blueprintId, enterpriseId, deployedAt)` — loads policies via `loadPolicies`, runs `evaluatePolicies`, upserts `deploymentHealth` row, returns previous + current status
- `checkAllDeployedAgents(enterpriseId)` — fetches all deployed blueprints for enterprise, calls `checkDeploymentHealth` for each, returns `{ checked, critical, results }`

### 6. `src/lib/monitoring/policy-impact-handler.ts` — Created

Side-effect module. Registers an event handler via `registerHandler`. On `policy.created`, `policy.updated`, or `policy.deleted`: calls `checkAllDeployedAgents`, then writes a `blueprint.health_checked` audit entry per result (actor: `"system"`). Triggers the notification handler's transition logic for compliance officer alerts.

### 7. `src/lib/notifications/handler.ts` — Modified

Extended `Meta` interface: added `healthStatus?`, `previousStatus?`, `errorCount?` fields. Added `blueprint.health_checked` event branch at end of `handleLifecycleEvent`: detects `clean→critical` (degraded) and `critical→clean` (restored) transitions; notifies all compliance officers with `deployment.health_degraded` or `deployment.health_restored` notification type.

### 8. `src/app/api/blueprints/[id]/status/route.ts` — Modified

Added import for `checkDeploymentHealth`. After the audit log write for `deployed` transitions, added fire-and-forget call: `void checkDeploymentHealth(id, blueprint.enterpriseId ?? null, new Date())`.

### 9. `src/app/api/monitor/route.ts` — Created (GET)

`requireAuth(["reviewer", "compliance_officer", "admin"])`. Fetches all deployed blueprints for enterprise using `selectDistinctOn` on `agentId` (latest version per logical agent). Batch-fetches health records via `inArray`. Returns `{ agents: AgentHealthEntry[], summary: { total, clean, critical, unknown } }`.

### 10. `src/app/api/monitor/[agentId]/check/route.ts` — Created (POST)

`requireAuth(["compliance_officer", "admin"])`. Fetches latest deployed blueprint for given `agentId` (enterprise-scoped, 404 if not found or not deployed). Calls `checkDeploymentHealth`. Writes `blueprint.health_checked` audit entry with `agentName` derived from `bp.abp.identity?.name ?? bp.name ?? "Unknown Agent"`. Returns `{ agentId, healthStatus, errorCount, warningCount, checkedAt }`.

### 11. `src/app/api/monitor/check-all/route.ts` — Created (POST)

`requireAuth(["compliance_officer", "admin"])`. Calls `checkAllDeployedAgents`. Writes audit entry per result. Returns `{ checked, critical }`.

### 12. `src/app/monitor/page.tsx` — Created

Client component. Data fetched from `GET /api/monitor` on mount; refetched after "Check All". Layout:
- 4 KPI cards: Total Deployed / Clean / Needs Attention / Not Checked
- Filter bar: text search (name / agentId) + health status dropdown
- Table: Agent | Version | Health | Errors | Warnings | Deployed | Last Checked | Actions
- Inline "Check Now" per row (compliance_officer/admin only) — updates row + recomputes summary via consolidated `setAgents` callback
- Health badges: green (clean), red (critical with error count), gray (not checked)
- "Check All Agents" button with spinner
- Empty state with link to `/deploy`
- `timeAgo` utility for relative timestamps

Fixed TypeScript error TS7053: consolidated `setAgents` + `setSummary` into a single callback to avoid `healthStatus` string indexing.
