# Session 157 — 2026-04-18

## Context

The Strategic Product Analysis v2 (this session's briefing) asked for the optimum next best actions to enable a credible end-to-end lifecycle demo with AWS AgentCore. The reality-check pass that preceded this session established that **7 of 8 lifecycle stages are code-complete**: intake (LLM-driven), ABP generation (Claude `generateObject`), governance validation (enforced on the `draft → in_review` gate), approval (multi-step chain + legacy single-step, SOD-enforced), deployment (real `BedrockAgentClient` calls in `src/lib/agentcore/deploy.ts`), monitoring (CloudWatch poller + `/api/telemetry/ingest`), and iteration (`new-version` endpoint). The one disconnected stage is **runtime execution**: Intellios has zero code paths that call `InvokeAgent` on a deployed Bedrock agent. Retirement is a status flag only — there is no corresponding `DeleteAgent` call, so deprecated blueprints orphan AWS resources indefinitely.

Positioning constraint: `docs/roadmap.md` explicitly frames Intellios as the "governed control plane" with execution on cloud provider runtimes. Adding an invocation path risks collapsing that positioning unless framed as a **governed test harness** rather than a runtime. That framing is the job of ADR-027.

Scope this session: close the runtime-execution gap with a reviewer-scoped Test Console, close the retirement gap with a real AgentCore de-registration call, and lock the strategic framing in an ADR. Defer one-time live smoke deploy (requires sandbox AWS creds) and demo rehearsal to follow-up sessions.

## Actions

1. Wrote **ADR-027 — Test Console as governed test harness**. Status: proposed. Frames the new `invokeAgent()` adapter and `/registry/[agentId]/test` UI as a reviewer+ scoped, rate-limited, audit-logged *test* surface — not a runtime. Cross-references ADR-010 (ABP → Bedrock translation). The ADR calls out the four guardrails that keep Test Console from becoming a runtime: (a) reviewer/compliance_officer/admin only — never architect or designer, (b) `max: 10` invocations per minute per actor, (c) every invocation writes a `blueprint.test_invoked` audit row with `agentId`, `sessionId`, and prompt hash, (d) the session id is client-generated and never persisted server-side.

2. Created `src/lib/agentcore/invoke.ts`. Thin adapter over `BedrockAgentRuntimeClient.InvokeAgentCommand` from `@aws-sdk/client-bedrock-agent-runtime`. Returns an `AsyncIterable<string>` of response chunks so the route can stream to the browser. Handles the three terminal conditions: agent not prepared (404 → `AGENT_NOT_DEPLOYED`), invocation error (5xx → `AGENTCORE_INVOKE_FAILED`), and RETURN_CONTROL responses (currently rendered as a synthetic "tool call simulated" chunk — real tool execution is out of scope for the Test Console per ADR-027).

3. Extended `src/lib/agentcore/deploy.ts` with `retireFromAgentCore(deployment, config)`. Calls `DeleteAgentCommand` with `skipResourceInUseCheck: true` and `GetAgentCommand` polling until the agent is gone or 30s elapses. Idempotent: a `ResourceNotFoundException` is treated as success (the agent is already gone). Returns `{ retired: true, retiredAt }` on success, throws on any other terminal error.

4. Extended `src/lib/agentcore/types.ts` with `AgentCoreRetirementRecord` (`target`, `agentId`, `retiredAt`, `retiredBy`) and added `retirement?: AgentCoreRetirementRecord` to `AgentCoreDeploymentRecord` so retirement evidence rides inside the same deployment blob.

5. Added two new error codes to `src/lib/errors.ts`: `AGENT_NOT_DEPLOYED` (409 — blueprint has no `deploymentMetadata.agentId` or is not in status `deployed`) and `AGENTCORE_INVOKE_FAILED` (502 — live `InvokeAgent` call failed).

6. Wrote `src/app/api/registry/[agentId]/invoke/route.ts`. Resolves the latest-deployed blueprint version under the given `agentId`, enforces `status === "deployed"` + `deploymentTarget === "agentcore"`, rate-limits via `rateLimit(email, { endpoint: "invoke", max: 10, windowMs: 60_000 })`, and streams the `InvokeAgent` response back as a `ReadableStream`. Writes a `blueprint.test_invoked` audit row pre-stream (non-blocking; audit failure logs but does not block the invoke). Dispatches a `blueprint.test_invoked` event post-stream.

7. Hooked `retireFromAgentCore()` into `PATCH /api/blueprints/[id]/status` on the `deprecated` transition. Best-effort — a retirement failure logs but does **not** block the status change, because the governance/audit semantics of deprecation must survive a Bedrock outage. On successful retirement, the deployment blob is merged with `retirement` metadata and written back to `deploymentMetadata`. The retire call is gated on `blueprint.deploymentTarget === "agentcore"` — blueprints deprecated without a live deployment (e.g. never-deployed drafts) skip the AWS round-trip entirely.

8. Built `src/app/registry/[agentId]/test/page.tsx` — the Test Console. Catalyst primitives (`Button`, `Heading`, `Input`, `Badge`, `Text`, `Divider`); reuses `src/components/chat/message-bubble.tsx` for the streamed bubble rendering. Session id is `crypto.randomUUID()` on first mount, persisted in React state only (never hits the server). Fetches agent metadata once from `/api/registry/[agentId]` to render a status badge and deployment target chip; refuses to render the prompt input if `status !== "deployed"`. Streams responses from `/api/registry/[agentId]/invoke` via `fetch().body.getReader()`.

9. Added `@aws-sdk/client-bedrock-agent-runtime ^3.1024.0` to `src/package.json`, matching the existing `@aws-sdk/client-bedrock-agent` version.

10. Wrote `docs/demo/lifecycle-demo.md` — 8-stage scripted walkthrough for the Retail Bank Customer-FAQ agent demo, with target timings, fallback paths for each stage, and a troubleshooting matrix keyed by the failure modes most likely to surface on the live path (Bedrock prep timeout, guardrail reject, credential drift, throttling, RETURN_CONTROL inconclusive response).

11. Updated `docs/roadmap.md` with a new "Phase 3 — Lifecycle Closure" section listing the invoke adapter, Test Console, retirement, one-time live smoke deploy, and demo runbook. Marked 157 complete; one-time live smoke deploy deferred to session 158 (needs sandbox AWS creds outside this sandbox's reach).

Updated `docs/decisions/_index.md` with the ADR-027 row; `docs/log/_index.md` with the session 157 row; `docs/log/effort-log.md` with the session 157 entry; `docs/project-journal.md` with the session 157 narrative; `docs/open-questions.md` with OQ on whether to build a RETURN_CONTROL tool-mock service for end-to-end tool exercising in Test Console.

Full vitest suite unchanged — no new behavior tests were added this session by design: the `invokeAgent()` adapter is thin enough that a mock-only test would test the SDK wiring rather than product behavior, and the real confidence signal is the one-time live smoke (session 158).

## Summary

Session 157 closes the runtime-execution and retirement gaps that were blocking a credible 8-stage lifecycle demo. The strategic framing — Test Console as a governed *test* harness, not a runtime — is locked in ADR-027.
