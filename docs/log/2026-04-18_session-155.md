# Session 155 — 2026-04-18

## Context

Fourth and final beat of the Production-Readiness-Review arc committed to in session 152: H3 (152) → H5 (153) → H6 (154) → **H4 (155)**. This session closes the outbound-webhook reliability finding and the PRR arc.

The 2026-04-17 PRR identified H4: ADR-009's outbound webhook path retried a failing delivery three times over ~3 seconds and then marked it `status='failed'` with no further action. A flapping subscriber whose 30-second recovery happened at second 4 would lose every event fired during the outage. There was no DLQ terminus, no operator visibility into stuck deliveries, no replay path, and — critically — no distinction between "the subscriber is briefly down" and "the subscriber has been 500ing for an hour." The PRR remediation note gestured at "queue + DLQ"; the actionable shape was ADR-026.

Goal: extend the retry window from 3 seconds to ~35 hours using scheduled cron-driven retries, introduce `status='dlq'` as an explicit terminus, add admin replay, and do it all without adding a worker deployment or a queue broker.

## Decisions

1. **Single inline attempt + scheduled retries, not in-process retry loop.** The old path did 3 retries in ~3 seconds while holding the inbound request's function instance and, critically, while the user was watching a spinner. The new path returns immediately after one attempt and lets cron carry the retry budget. The inline attempt is still worth keeping because ~90% of webhooks in healthy operation succeed on first try, and the inline path lets the admin UI render "delivered" synchronously in those cases without a round-trip to the DB.
2. **Hardcoded `BACKOFF_SCHEDULE_MS` array, not an algorithmic formula.** Operators should be able to cross-reference a row's `attempts=N, next_attempt_at=X` against a literal list of durations and know why X was chosen. An algorithmic base × 2^n formula with a floor and a ceiling is the same six numbers, but now obfuscated behind three configuration knobs that nobody will know the current values of at 3am. The hardcoded array is dumber and more legible.
3. **Reuse existing `webhook_deliveries` table; add `dlq` as a new `status` value.** A separate `webhook_dead_letters` table was the first instinct — it's how you'd build it if the webhook_deliveries table were large and dlq scans were a real cost. At current scale, the table is small and `status='dlq'` with a partial index on `status='pending'` keeps the cron scan tight while the admin UI's "DLQ view" is just one filter away from the "Recent deliveries" view it's already rendering.
4. **`max_attempts` is a per-row column with default 7, not a global constant.** Most webhooks get the default. The column exists so that the next time a specific subscriber is known-flaky but important (think: billing partner with a known 2-hour weekly maintenance window), an operator can bump *that* webhook's budget to 10 without a schema change or a code deploy. The `BACKOFF_SCHEDULE_MS` array clamps to its last entry if `max_attempts > schedule.length`, so the override works without a corresponding schedule edit.
5. **`deliverWebhookTest` stays single-attempt (`maxAttempts: 1`).** The admin "Test" button exists to render a single deterministic verdict — it's a diagnostic, not a scheduling action. Routing it through the retry machinery would mean a 35-hour-delayed test, which defeats the point. Side effect: test deliveries that fail land in `dlq` immediately instead of `failed`; the admin UI should label test deliveries distinctly. This is a follow-up.
6. **408 and 429 are retryable despite being 4xx.** Both are "the server wants you to try again later" semantics baked into HTTP. Treating them as non-retryable would send a transient subscriber-side rate-limit straight to DLQ, which is hostile to correct subscribers. Everything else in 4xx stays non-retryable because it's the subscriber telling us we're wrong.
7. **Replay preserves the original `deliveryId`, not a new one.** Subscribers deduplicate on the `X-Intellios-Delivery` header. Generating a fresh UUID on replay would guarantee double-processing on the subscriber side for any replay where the original attempt had in fact succeeded but been recorded as failed (subscriber ack swallowed, etc.). Preserving the ID is the safe default; if a subscriber genuinely needs a distinct event for a replay, a future enhancement can surface a `forceNewId=true` option.
8. **Replay refuses when the parent webhook is inactive.** The retry cron would pick up the delivery, find the webhook deactivated, and flush it to DLQ as `webhook_inactive`. That round-trip is pure noise — return a 422 synchronously instead and tell the admin to reactivate first. The audit trail stays clean: one replay event → one actual attempt, not one replay event → one cron failure → one DLQ entry.

## Deferred / follow-ups

- Admin UI "DLQ count" badge on the webhooks list + a DLQ-filtered delivery view. The backend supports it today; the UI wire-up is a cosmetic session.
- Alerting on DLQ growth rate (e.g., `> 10 new DLQ entries per enterprise per hour`). Belongs alongside the telemetry/alerts work, not the retry path.
- Per-subscriber circuit breaker — if one webhook is 500ing on 90% of deliveries, the retry cron still wastes budget on it. ADR-023 pattern for Bedrock adapts cleanly. Not urgent at current scale.
- Capture a rolling window of response bodies per delivery (not just the last one) so DLQ investigation has actual payload history. Requires a new table or a JSONB column; session-sized.
- Label test deliveries distinctly in the admin UI — currently `deliverWebhookTest` failures land in DLQ like any other failure, which is correct behavior but presentationally surprising. See Decision #5.
- `POST /api/admin/webhooks/deliveries/[id]/cancel` — force a pending delivery to DLQ without replay. Useful when an admin knows a specific event is irrelevant to retry (e.g., user was deleted between fire and retry). Not blocking; one-session addition.

## Summary

The outbound webhook path is no longer a 3-second window. A failing subscriber now gets ~35 hours of scheduled retry with `[60s, 5m, 30m, 2h, 8h, 24h]` jittered backoff, driven by a per-minute cron that reuses ADR-024's `runCronBatch` for time-budget safety and per-item isolation. Terminal failures land in a new `status='dlq'` terminus instead of vanishing into `status='failed'`. Admin operators can replay any delivery — including successful ones — via `POST /api/admin/webhooks/deliveries/[id]/replay`, which audits the replay with prior-state metadata and preserves the original delivery ID so subscriber-side dedupe still works.

Test surface grew from 661 → 711 (+50 across 3 new test files and 1 modified). Typecheck unchanged — 7 pre-existing errors in `blueprint-lifecycle.test.ts`, zero new.

PRR arc closed. H3, H5, H6, H4 — four beats, four sessions, four ADRs (023, 024, 025, 026). No remaining High-severity items from the 2026-04-17 review.
