## 4. Gate 1 Shallow-Clone Finding — Third Workaround, SCRUM-34 Filed

With the dirty-state items resolved, the push went through. The resulting Vercel build (`dpl_5U6WmXPmHByFN2ban76tYeD9yTTe`) was CANCELED by Gate 1. Build log:

```
[ignore-build] VERCEL_GIT_PREVIOUS_SHA=dd12b6ce unreachable after fetch; falling back to HEAD~1.
[ignore-build] No changes in src/ since HEAD~1 — skipping build.
```

ADR-028's Gate 1 fix uses `VERCEL_GIT_PREVIOUS_SHA` as the diff baseline, with `git fetch --depth=200 origin <SHA>` as a fallback for shallow clones. The gap: the targeted fetch fails when the gap is 12+ commits (dd12b6c was 12 commits behind the push tip). Gate 1 fell back to HEAD~1, saw a docs-only commit at the tip (`d05f629`), and skipped the build. The three session-166 src/ bugfixes in `9bbe770` remained undeployed.

This was the **third use** of the version-bump workaround. Sessions 158 and 159 had already established this pattern; ADR-028 was authored in session 159 specifically to retire it. The fallback logic in `scripts/vercel-ignore-build-step.sh` has a depth limit that the implementation did not account for.

**SCRUM-34** was filed (To Do, Medium, labels: `sys:dx`, `concern:ci`): "Gate 1 shallow-clone fallback exceeds depth — upgrade to `git fetch --unshallow`." The long-term fix replaces the targeted depth-limited fetch with a progressively-deepening retry or an `--unshallow` call, and adds a synthetic harness test for the deep-gap scenario.

**Commit:** `922591a` — `chore(release): bump version 0.1.2 → 0.1.3 to trigger Gate 1`

The version bump touches `src/package.json`, making HEAD~1 see a real `src/` change. Gate 1 passed on the next push attempt.

## 5. Linux Production Regression — SCRUM-35 Filed, Resolved in c6308a3

The version-bump push triggered Vercel build `dpl_6T9pJxNeAmxKRswPUY6ykHSQmZA3`, which ERRORed with four identical module-resolution failures:

```
Module not found: Can't resolve './vercel/path0/src/node_modules/@react-aria/focus/dist/main.js'
Error: server relative imports are not implemented yet.
```

**Root cause:** `9bbe770`'s `next.config.ts` fix added `resolveAlias` entries using `path.resolve()` to pin `@react-aria/*` to compiled CJS output. On Windows, `path.resolve()` returns `C:/...` paths; Turbopack correctly identifies these as filesystem paths. On Linux (Vercel's production environment), `path.resolve()` returns `/vercel/path0/src/...` — paths starting with `/`. Turbopack's module resolver interprets a leading `/` as a server-relative URL, not a filesystem path. The fix was Windows-correct and Linux-broken.

The last clean Linux production build was `dd12b6c` (session 162), which had no `resolveAlias` entries at all. The `@react-aria` source-condition issue that motivated the Windows fix does not occur on Linux (Turbopack does not pick up the `source` export condition in the Linux environment).

**SCRUM-35** was filed (To Do, Medium, labels: `sys:dx`, `concern:build`, Epic SCRUM-25): "next.config.ts: guard Turbopack resolveAlias @react-aria entries to Windows only."

**Fix:** Wrap the entire `resolveAlias` block in `process.platform === "win32"`. On Linux the block is absent, reverting to `dd12b6c`'s behavior. On Windows dev, the existing fix is preserved. The `replace(/\\/g, "/")` backslash normalization inside the block remains correct (Windows paths in Windows strings).

**Commit:** `c6308a3` — `fix(next-config): guard resolveAlias @react-aria entries to Windows only`

**Result:** Vercel build `dpl_DAT9vCjBkEG3uGdzQiZhq7XBUr41` — READY. Production shifted from `dd12b6c` to `c6308a3`. All three session-166 bugfixes (`9bbe770`: `translate.ts` agent-name sanitization, `deploy.ts` CREATING-state race, `next.config.ts` Turbopack Windows paths) are serving production.

**SCRUM-35** transitioned Done at session close (comment 10159, transition ID 51).

The long-term fix (replace `resolveAlias` filesystem paths with `turbopack.resolveConditions` to drop the `source` export condition globally, removing hardcoded paths entirely) is tracked in **SCRUM-36** (To Do, Low, labels: `sys:dx`, `concern:build`).
