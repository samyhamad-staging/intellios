## Key Decisions

- **Audit badge is purely informational, not interactive** — The firstMessage flag that triggers the audit entry is already wired into the transport. The badge just surfaces the existing behavior so users know it's happening. Green + ShieldCheck = trust signal (not a warning).
- **Violations sparkline from in-memory data** — No new API call needed: `violations` array is already loaded. The bucket computation groups by `detectedAt.slice(0, 10)` (YYYY-MM-DD) which is O(n × days) but n and days are both small (max 100 violations × 30 days). 1d range excluded because 1 day = 1 possible bar, which doesn't convey trend.
- **Dashboard next actions via `/api/me`** — The dashboard is a `"use client"` component with no server-side user context. `/api/me` is a lightweight endpoint already used elsewhere (review-panel SOD check). `Promise.all` with the existing summary fetch avoids a sequential waterfall — both fire in parallel. The `userRole` state starts null and the Actions section only renders once both calls complete.
- **Urgent state on reviewer/governor cards** — When `inReview > 0`, the "Awaiting review" card turns amber to signal actionable urgency. This is the most impactful signal for reviewers who open the dashboard and need immediate direction.
