## Actions

### Phase 13: Governance Integrity

#### 1. `src/lib/governance/types.ts` — Modified
Added `evaluatedPolicyIds: string[]` to `ValidationReport`. Records which policy IDs were evaluated in each validation run, enabling:
- Audit trail evidence of exactly which policies a blueprint was validated against
- Future staleness detection (compare set against current live policies)
- MRM report traceability (report can show which policies were active at approval time)

#### 2. `src/lib/governance/validator.ts` — Modified
Updated `validateBlueprint` to include `evaluatedPolicyIds: policies.map(p => p.id)` in the returned report. Both the zero-policy early-return and the full evaluation path now return the field.

#### 3. `src/app/api/blueprints/[id]/status/route.ts` — Modified
**Core P0 fix.** Replaced the stale-report check in the `in_review` transition with live revalidation:

**Before (stale):**
```
if newStatus === "in_review":
  read blueprint.validationReport (stored, potentially stale)
  if report && report.violations.filter(error).length > 0: block
  else: allow through — even if policies changed since report was run
```

**After (live):**
```
if newStatus === "in_review":
  freshReport = await validateBlueprint(blueprint.abp, blueprint.enterpriseId)
  persist freshReport to DB (keeps workbench + MRM report current)
  if freshReport.violations.filter(error).length > 0: block with violations
  else: proceed
```

This guarantees that no blueprint can enter the review queue with unresolved governance violations against the **current** policy set. Policy changes between a designer's last manual validation and their submission are now caught automatically.

Added `governanceRevalidatedAt` timestamp to the `in_review` audit log metadata for compliance evidence.

Changed imports: replaced `ValidationReport` type import with `validateBlueprint` function + `ABP` type.

#### 4. `src/app/blueprints/[id]/page.tsx` — Modified
Added `reportIsFresh: boolean` state (initially `false`) to track whether the current validation report was:
- Loaded from the DB / URL params → `false` (may not reflect current policies)
- Just run this session via "Validate now" or "Re-run" → `true`

When `reportIsFresh === false` and a report exists, renders an amber warning strip below the validation status badge:
> "⚠ Saved from a previous session — re-validate to check against current policies before submitting."

`reportIsFresh` is set to `true` after a successful `handleValidate` call and reset to `false` on `handleRefine` (report cleared anyway, but state is also reset for consistency).
