# Session 144 — 2026-04-08

**SOD Enforcement Gap Fix — Legacy Single-Step Approval Mode**

## Context

Session 141 (blueprint lifecycle tests) discovered that the legacy single-step approval path in `PATCH /api/blueprints/[id]/status` did not enforce Separation of Duties (creator ≠ approver). The multi-step chain path correctly enforced SOD, but when an enterprise has no approval chain configured (the default), a blueprint creator could approve their own work. This is a governance integrity gap.

## Actions

### 1. Analyzed the SOD enforcement gap

Read the status route (`src/app/api/blueprints/[id]/status/route.ts`) and review route (`src/app/api/blueprints/[id]/review/route.ts`) to understand the two paths to approval:

- **Status route (legacy):** `PATCH /api/blueprints/[id]/status` with `{ status: "approved" }`. When `approvalChain.length === 0`, the route transitions directly to approved without checking `createdBy === userEmail`.
- **Status route (multi-step):** When `approvalChain.length > 0`, the route checks both `existingApprovers.includes(userEmail)` and `createdBy === userEmail`.
- **Review route:** Always checks `createdBy === userEmail` regardless of chain length.

The asymmetry meant a second path to approval existed that bypassed SOD.

### 2. Designed the fix

Hoisted the SOD check above the `if (chain.length > 0)` branch so it applies to **both** legacy single-step and multi-step modes. The multi-step block retains its additional `existingApprovers` check (preventing the same person from approving at multiple steps in a chain — a chain-specific concern).

The SOD check reads `settings.governance.allowSelfApproval` and, when false (default), rejects any approval where `blueprint.createdBy === userEmail`.

### 3. Implemented the fix

Modified `src/app/api/blueprints/[id]/status/route.ts`:
- Moved the `getEnterpriseSettings()` call and `allowSelfApproval` check to execute before the chain-length branch
- Returns 403 with "Self-approval is not permitted (SOD requirement)" when blocked
- Preserves the `allowSelfApproval: true` opt-out for enterprises that explicitly allow it

### 4. Updated tests

Modified `src/__tests__/blueprints/blueprint-lifecycle.test.ts`:
- Changed existing legacy SOD test from expecting 200 (documenting the gap) to expecting 403 (proving the fix)
- Added 2 new test cases in a "SOD enforcement (legacy single-step)" describe block:
  - Blocks self-approval when `createdBy === reviewer` in legacy mode (403)
  - Allows self-approval when `allowSelfApproval: true` (200)
  - Allows approval when reviewer is not the creator (200)
- Test count: 56 → 58

### 5. Verified all tests pass

Ran vitest: 58/58 blueprint lifecycle tests pass.

## Findings

- The SOD gap was in production since the status route was first implemented. The review route always enforced SOD, so enterprises using the review workflow were unaffected. Only enterprises using the direct status-transition API for approval were exposed.
- The `allowSelfApproval` setting was already wired into the multi-step path but not the legacy path — the setting existed, its enforcement was just incomplete.

## Summary

Fixed a governance integrity gap where legacy single-step blueprint approval did not enforce Separation of Duties. The creator of a blueprint can no longer approve their own work unless the enterprise explicitly opts out via `allowSelfApproval`. Added 2 new test cases and modified 1 existing case to prove the fix. Total test count: 554.
