## Key Decisions

- **CompanionChat was defined but never rendered** — the component existed in the codebase with a working `onApply` callback but was not imported anywhere. The fix was to import it into the blueprint page and give it a dedicated "Companion AI" tab in the right rail. Switching it to the Refine tab on Apply gives the architect immediate visual confirmation that the change was queued.
- **`handleRefine(directPrompt?)` not a new function** — rather than creating a separate `handleApplyCompanionChange` function, I extended the existing `handleRefine` to accept an optional prompt. This keeps the refinement logic in one place and avoids drift between the two code paths.
- **useChat `initialMessages` for history** — AI SDK v6 (`@ai-sdk/react` v3) supports `initialMessages: UIMessage[]` on the `useChat` hook. Passing messages loaded from localStorage gives the returning architect their full conversation context without any backend changes.
- **Clear = reload** — `useChat` state lives inside the hook's closure; there is no public reset API. Removing from localStorage + `window.location.reload()` is the honest solution. The alternative (key-based remount) would require lifting state to the parent, which is more invasive. The reload is infrequent and intentional.
- **Violations acknowledgement is browser-local** — The UX map asks for acknowledgement to distinguish "known/accepted" violations from new ones. A DB migration would be the correct long-term fix (`acknowledged_at` + `acknowledged_by` columns). Client-side localStorage avoids a migration while delivering the core UX value: acknowledged violations dim and can be mentally filtered. The limitation (not shared across browsers) is acceptable for the current sprint.
