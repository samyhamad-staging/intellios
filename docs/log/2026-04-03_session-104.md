## Key Decisions

- **Click-to-send for saved scenarios, not click-to-insert** — `ChatInput` manages its own `value` state internally with no external control prop. Inserting into the input would require either (a) modifying `ChatInput` to accept a controlled value prop, or (b) lifting state to `SimulatePanelInner`. Both are more invasive than the zero-dep constraint permits for a P1 item. Click-to-send (calling `handleSend(scenario.text)`) is pragmatic, matches the typical "quick prompt" UX pattern in AI playgrounds, and keeps the code change surgical.
- **Inline add form, not a modal** — The scenario library is a compact power-user feature. An inline form (expands in place) avoids a modal that would cover the chat history and is consistent with the add-scenario UX in tools like Postman collections.
- **Monitor acknowledgement mirrors violations pattern** — P1-31 (violations acknowledgement) and P1-33 (monitor alert acknowledgement) use the same pattern: `Set<string>` in state + localStorage, green chip in header count row, opacity-60 on acked rows, toggle button. This creates a consistent mental model: "acknowledged" means the same thing whether you're looking at a violation or a degraded agent.
- **`isAlertable` guards Acknowledge button** — Only `degraded` and `critical` agents show the button. `clean` and `unknown` agents don't surface alerts, so the acknowledgement workflow doesn't apply to them.
