## Context

Session 152 shipped H3 (Bedrock circuit breaker). The 2026-04-17 Production-Readiness Review flagged H5 next: six Vercel cron routes (`alert-check`, `governance-drift`, `portfolio-snapshot`, `quality-trends`, `review-reminders`, `telemetry-sync`) all iterate over tenants/agents/blueprints inline with three concrete defects:

1. **No time-budget guard** ‚Äî a slow enterprise in a nightly loop can consume the full 60s function cap, and the affected enterprise never recovers until the next run attempts it again from the top.
2. **Inconsistent per-item isolation** ‚Äî `review-reminders` had zero `try/catch` around its per-blueprint work; one reminder throwing would abort the entire loop. The others had loose `console.error` without structured persistence.
3. **No partial-completion memory** ‚Äî if a run fails halfway, the next invocation starts from the same order and hits the same stuck items first, producing a cascading backlog.

The PRR's remediation note suggested "cron job queue (BullMQ/SQS)." That was rejected in favour of a DB-backed batch runner + priority-reorder pattern that delivers ~95% of the operational value with zero new infrastructure and stays consistent with ADR-023's rejection of Redis on the failure path.

## Decisions Made

1. **DB-backed batch runner, not BullMQ/SQS.** The PRR remediation note called for a "cron job queue." A queue solves the problem but brings its own operational surface: BullMQ wants Redis, SQS wants IAM + networking, both want a worker deployment separate from Vercel Functions. The failure mode H5 actually guards against ‚Äî "one slow tenant poisons the whole nightly run" ‚Äî doesn't need durable cross-instance queuing; it needs a per-invocation time-budget guard + per-item failure isolation + memory of what failed so the next invocation prioritizes it. Two tables and a generic runner delivers that with zero new infra. ADR-024 documents the "revisit when tenant count > ~200" trigger for reconsidering the queue path.

2. **No within-run retry.** The runner exits to the next scheduled invocation on item failure. Within-run retry would double-charge the time budget for the same item when the underlying problem is persistent (a stuck Bedrock, a pg_connection saturation), and transient failures are better handled by the next cron tick with a fresh budget envelope. Documented in ADR-024's "Consequences ‚Äî refuses-to-retry" section.

3. **Budget-exhausted runs return 200, not 500.** The cron made forward progress within its guaranteed envelope; the skipped items are recorded and will be prioritized on the next invocation. Returning 500 would misclassify a healthy-but-slow run as an operational failure, flood the on-call alert channel, and teach the team to ignore cron errors.

4. **Priority reorder via DB query, not in-memory scratchpad.** `recentFailedItemIds(jobName, 48h)` queries `cron_item_failures` on every run start. This is one extra query per cron invocation (cheap; indexed) and it survives process restarts, function recycling, and regional failovers. An in-memory cache would be faster and strictly incorrect for a multi-instance Vercel deployment where the "next run" may land on a different instance than the failed one.

5. **Failure classification: any thrown error counts.** Unlike the Bedrock breaker (ADR-023), which skips 4xx, the batch runner persists every thrown error. The cron loop's caller is infrastructure, not an end-user, and "this agent's evaluation threw a ValidationError" is just as worth surfacing in `cron_item_failures` as a 5xx. `String(err)` is used for non-Error throws so the failure row is always populated.

6. **Asymmetry between runner-scoped and handler-scoped state.** The runner counts `succeeded`/`failed`/`skipped` generically; job-specific counters (`regressions`, `drifted`, `cleared`, `sent`, `checked`, `breached`, `synced`) live in the route's enclosing scope and are closed over by the handler. This keeps the runner generic (no per-job counter type parameter) and lets each route report its natural metrics without a schema-side dictionary.

7. **Recipient email list loaded once per `checkAndFireAlerts` batch, not per agent.** Pre-batch optimization. The recipient set is enterprise-scoped and doesn't change mid-run; loading it inside the handler would issue N redundant queries per cron tick.

## Deferred / Follow-up

- **DLQ browse UI.** Operators will need a way to inspect `cron_item_failures` without raw SQL. Deferred to a future session. The table is there, indexed, and FK-cascaded; the UI is additive.
- **Cron-run backlog alerting.** If a job reports `budget_exhausted=true` on N consecutive runs, the backlog is growing faster than the schedule can drain it. Not in scope for H5. Would layer on top of the observability floor from session 149.
- **Per-item retry counter with backoff on the priority queue.** An item that has failed 10 runs in a row probably needs a human, not another priority boost. Deferred ‚Äî rare in practice; would require a schema extension and a de-prioritization tier.

## Summary

Session 153 closed the largest cron-reliability gap in the production-readiness review: a single slow or broken tenant in a nightly loop can no longer poison the rest of the run, and the system now remembers what it didn't finish so the next invocation starts with the stuck items at the front. The `runCronBatch<T>` generic + two DB tables replaced six hand-rolled loops across the codebase, each of which had slightly different failure-handling conventions (and one of which ‚Äî `review-reminders` ‚Äî had none at all). The priority-reorder pattern (`recentFailedItemIds` ‚Üí `prioritizeFailed`) means transient-but-persistent failures drain faster on the next run instead of slower, and the time-budget guard means the Vercel 60s cap stops being load-bearing on the tail-latency of one agent. 13 new tests, 600/600 green, typecheck clean on all new code. The H3 ‚Üí H5 arc is half-closed; H6 and H4 remain for sessions 154 and 155.
