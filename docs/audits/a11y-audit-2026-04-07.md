## Detailed Findings

### A-01: `text-text-tertiary` Contrast at Small Sizes

**WCAG Criterion:** 1.4.3 Contrast (Minimum)
**Severity:** Major
**Locations:** `src/app/page.tsx` (lines 127, 229, 297, 540+), `src/app/admin/users/page.tsx`, `src/app/admin/api-keys/page.tsx` (line 199–209), various timestamp/metadata labels throughout

**Issue:** The design token `text-text-tertiary` resolves to approximately slate-400 (#94a3b8) in light mode. Against a white or light surface background, this achieves roughly 4.1:1 contrast — which passes for large text (18pt+) but fails the 4.5:1 threshold for normal text. Many instances appear at `text-xs` or `text-sm` size, which is normal text.

**Recommended Fix:**
- For small text (text-xs, text-sm, text-base): use `text-text-secondary` (slate-500, ~5.5:1)
- Reserve `text-text-tertiary` for large text (text-lg+) or icon coloring only
- Alternatively, darken the `--color-text-tertiary` token to achieve 4.5:1

### A-02: Opacity-Modified Red Text

**WCAG Criterion:** 1.4.3 Contrast (Minimum)
**Severity:** Major
**Location:** `src/app/page.tsx` line 540 — `text-red-500/60`

**Issue:** Tailwind's `/60` opacity modifier reduces red-500 (#ef4444) to roughly 60% opacity, yielding an effective contrast of ~2.8:1 on light backgrounds. This fails both AA thresholds.

**Recommended Fix:** Replace `text-red-500/60` with `text-red-700` or the semantic `text-danger` token, which maintains full opacity and achieves >4.5:1.

### A-03: Placeholder Text Contrast

**WCAG Criterion:** 1.4.3 Contrast (Minimum)
**Severity:** Minor
**Location:** `src/components/catalyst/input.tsx` line 75 — `placeholder:text-text-tertiary`

**Issue:** Placeholder text uses the same tertiary token. WCAG 1.4.3 technically applies only to "text," and placeholder text is advisory. However, WCAG 2.2 guidance and best practice recommend meeting 4.5:1 for placeholders since users rely on them.

**Recommended Fix:** Use `placeholder:text-text-secondary` or darken the tertiary token.

### A-04: Missing `scope="col"` on Table Headers

**WCAG Criterion:** 1.3.1 Info and Relationships
**Severity:** Major
**Location:** `src/components/catalyst/table.tsx` lines 77–90

**Issue:** The Catalyst `TableHeader` component renders `<th>` elements without `scope="col"`. Screen readers cannot programmatically associate header cells with data cells, degrading the table experience for assistive technology users. Session 124 notes mention this fix but it is not present in the current code.

**Recommended Fix:**
```tsx
export function TableHeader({ className, ...props }: React.ComponentPropsWithoutRef<'th'>) {
  return (
    <th
      {...props}
      scope="col"
      className={clsx(className, /* existing classes */)}
    />
  )
}
```

### A-05: Decorative Icons Missing `aria-hidden`

**WCAG Criterion:** 4.1.2 Name, Role, Value
**Severity:** Major
**Locations:** Throughout `src/app/page.tsx`, `src/app/admin/*.tsx`, `src/app/review/page.tsx`, `src/app/deploy/page.tsx`, `src/components/ui/empty-state.tsx`

**Issue:** Lucide React icons used as visual decoration (next to text labels, in cards, in headings) are announced by screen readers as meaningless content. Example: `<ChevronRight size={14} />` inside a card is read aloud despite conveying no information beyond the text.

**Recommended Fix:** Add `aria-hidden="true"` to all decorative icons:
```tsx
<ChevronRight size={14} aria-hidden="true" />
```
For icon-only interactive elements, keep the icon visible to assistive tech but add `aria-label` on the parent button (see A-06).

### A-06: Icon-Only Buttons Without `aria-label`

**WCAG Criterion:** 4.1.2 Name, Role, Value
**Severity:** Major
**Locations:** Various close/dismiss buttons, filter toggles, the copy button in `src/app/admin/api-keys/page.tsx` (line 117), revoke buttons

**Issue:** Buttons that contain only an icon and no visible text have no accessible name. Screen readers announce them as "button" with no context.

**Good Example Already Present:** `src/app/admin/api-keys/page.tsx` line 217 correctly includes `aria-label="Revoke key"`.

**Recommended Fix:** Audit all icon-only buttons and ensure each has `aria-label`:
```tsx
<button aria-label="Copy to clipboard" onClick={handleCopy}>
  <Copy className="h-4 w-4" aria-hidden="true" />
</button>
```

### A-07: No `prefers-reduced-motion` for Custom Animations

**WCAG Criterion:** 2.3.3 Animation from Interactions (AAA, but good AA practice)
**Severity:** Major
**Locations:** `src/app/globals.css` lines 364–378 (domain-scan, signal-bar, score-ring-circle), lines 468–474 (hero-float)

**Issue:** Four custom CSS animations run infinitely with no respect for `prefers-reduced-motion`. Users who experience vestibular discomfort cannot disable these.

**Recommended Fix:** Add to `globals.css`:
```css
@media (prefers-reduced-motion: reduce) {
  .hero-bg-float,
  [class*="domain-scan"],
  [class*="signal-bar"] {
    animation: none !important;
  }
  .score-ring-circle {
    transition: none !important;
  }
}
```
Note: Tailwind v4's built-in `animate-pulse` already respects `prefers-reduced-motion`.

### A-08: Inline SVGs Without Role/aria-hidden

**WCAG Criterion:** 1.1.1 Non-text Content
**Severity:** Minor
**Location:** `src/app/page.tsx` (hero section SVG backgrounds), `src/app/login/page.tsx` (logo SVG)

**Issue:** Inline `<svg>` elements lack either `role="img" aria-label="..."` (for meaningful SVGs) or `aria-hidden="true"` (for decorative SVGs).

**Recommended Fix:** Add `aria-hidden="true"` to decorative background SVGs. Add `role="img" aria-label="Intellios logo"` to the brand logo SVG.

### A-09: Inconsistent Form Error Associations

**WCAG Criterion:** 3.3.1 Error Identification
**Severity:** Minor
**Locations:** Pages using inline inputs outside `FormField` wrapper (e.g., `src/app/admin/integrations/page.tsx` InputField helper, search inputs)

**Issue:** The `FormField` component correctly generates `aria-describedby` IDs and `role="alert"` on errors, but not all form inputs in the codebase use `FormField`. The inline `InputField` in integrations/page.tsx (line 198) wraps with `FormField` (good), but standalone search inputs elsewhere may lack error association.

**Recommended Fix:** Audit all `<input>` elements and ensure either a `FormField` wrapper or manual `aria-describedby` / `aria-label` is present.

---

### A-10: Dialog Focus Restoration

**WCAG Criterion:** 2.4.7 Focus Visible / 2.4.3 Focus Order
**Severity:** Minor
**Location:** `src/components/catalyst/dialog.tsx` (Headless UI Dialog)

**Issue:** Headless UI's Dialog component should handle focus trapping and restoration automatically. However, custom dialogs or confirmation modals (e.g., the revoke-key dialog in api-keys) should be verified to ensure focus returns to the trigger button after close.

**Recommended Fix:** Manual browser testing with keyboard-only navigation. If focus is not restored, add `initialFocus` ref and `onClose` focus management.

### A-11: Badge/Status Color Contrast on Light Backgrounds

**WCAG Criterion:** 1.4.3 Contrast (Minimum)
**Severity:** Minor
**Locations:** `src/app/admin/users/page.tsx` (role badges), `src/app/page.tsx` (status indicators), `src/lib/status-theme.ts` (centralized status colors)

**Issue:** Some badge combinations (e.g., amber text on amber-50 background) may approach but not clearly pass the 4.5:1 boundary. The centralized `status-theme.ts` helps consistency, but specific dark mode / light mode combinations need verification with a contrast checker.

**Recommended Fix:** Run each badge variant through WebAIM's contrast checker in both themes. Document results and adjust any failing pairs.

## Strengths (No Action Needed)

**Skip Link:** Correctly implemented in `layout.tsx` with `sr-only focus:not-sr-only`, high z-index, and `#main-content` target. Excellent.

**Touch Targets:** Catalyst Button includes a `TouchTarget` wrapper ensuring 44×44px minimum hit area (button.tsx lines 194–204). Exceeds WCAG 2.2 AA 24px requirement.

**Skeleton Loading Accessibility:** `SkeletonList` uses `role="status"`, `aria-live="polite"`, and `aria-busy="true"`. Screen readers are properly informed of loading state.

**Error Announcements:** Login form error messages use `role="alert"` for immediate screen reader announcement. FormField errors use the same pattern.

**Semantic HTML:** Catalyst components use native HTML elements (`<button>`, `<table>`, `<input>`, `<dialog>` via Headless UI) rather than `div`-based replacements.

**Heading Hierarchy:** Pages consistently use Catalyst `Heading` component with appropriate `level` props (1 for page titles, 2 for sections).

**Keyboard Navigation:** Headless UI components (Dialog, Dropdown, Switch, Combobox) provide built-in keyboard interaction. No custom keyboard handling needed.

**Focus Indicators:** All interactive elements include `focus:outline-hidden data-focus:outline-2 data-focus:outline-offset-2` or similar visible focus styles.

**Dark Mode Design Tokens:** Color tokens are semantic (`--color-text`, `--color-surface`, `--color-border`) and adjust automatically for dark mode, maintaining intended contrast relationships.

## Testing Recommendations

**Automated:** Run `axe-core` or Lighthouse accessibility audit against all critical routes. Focus on contrast ratios, missing labels, and ARIA attribute validation.

**Keyboard:** Tab through every page end-to-end. Confirm all interactive elements receive visible focus and are operable. Test modal open/close focus flow.

**Screen Reader:** Test with VoiceOver (macOS/iOS) and NVDA (Windows). Verify table headers announce correctly, form errors are announced, and decorative icons are silent.

**Motion:** Enable `prefers-reduced-motion: reduce` in OS settings and confirm all animations stop.

**Zoom:** Test at 200% browser zoom — verify no content is clipped or hidden.
